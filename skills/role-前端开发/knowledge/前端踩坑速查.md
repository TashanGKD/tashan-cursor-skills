## 踩坑速查表

| # | 症状 | 根因 | 修复方案 |
|---|---|---|---|
| FE-01 | API 请求静默失败，控制台报 "Unexpected end of JSON input" | vite proxy 端口与后端 BACKEND_PORT 不一致 | 检查 vite.config.ts 中 proxy target 端口，与 .env 中 BACKEND_PORT 完全一致 |

| FE-04 | 登录后看不到应有的引导（如 AI 引导页），直接跳到聊天 | 根路由 `/` 固定跳转 `/chat`，未判断用户当前状态 | 实现 `HomeRedirect` 组件，检查用户是否有活跃组织，按状态跳转 |
| FE-05 | dev_code 一闪而过，用户来不及复制 | 没有给 dev_code 展示设置足够时间 | dev_code 展示计时器设为 15 秒，显示倒计时 |
| FE-06 | SSE 连接建立成功但收不到内容 | 用了 EventSource（不支持自定义请求头，无法带 Authorization） | 改用 fetch + ReadableStream 方式消费 SSE |
| FE-07 | TypeScript 类型错误导致构建失败 | API 返回类型与前端定义不一致 | 在 `src/types/` 维护与后端 Pydantic Schema 对应的 TypeScript 类型定义 |
| FE-08 | 页面加载慢，白屏时间长 | 没有代码分割，所有组件打进一个 bundle | 使用 `React.lazy + Suspense` 做路由级懒加载 |
| FE-09 | 浏览器访问 `http://localhost:3000` 报 Error -102 (ERR_CONNECTION_REFUSED) | Node 18+ 在 Windows 上默认将 `localhost` 绑定到 `::1`（IPv6），Chrome 先尝试 IPv4 `127.0.0.1` → TCP 被拒绝 | 在 `vite.config.ts` 的 `server` 节加 `host: '127.0.0.1'` 强制 IPv4 绑定；或直接访问 `http://127.0.0.1:3000/` |
| FE-10 | 浏览器控制台报 `ReferenceError: xxx is not defined`，功能静默失效 | 新增了对某函数的调用，但没有更新 `import` 语句。TypeScript 编写时不报错，运行时才暴露。常见场景：新增 SSE 事件处理、新增 API 调用、重构时移动了函数所在文件 | **写完任何新的函数调用后立即检查 import**：`Ctrl+F` 搜索函数名，确认 import 行存在。自测清单必须包含「新增调用的函数 import 已更新」|
| FE-11 | Vite 报 `[PARSE_ERROR] Unexpected token`，指向一行孤立的 `/>` 或 `)}` | 使用 StrReplace 删除代码时，锚点范围不足，只删除了代码块的开头/中间，结尾的 JSX 关闭符（`/>` `)}` `</div>`）被遗留为游离碎片 | **StrReplace 删除代码时，锚点必须覆盖完整的代码块**：从开头到最后一个关闭符（含 `)`）。删除后立即在浏览器确认无语法错误 |
| FE-12 | 切换视图（Tab/路由）后再切回，对话历史/表单输入/滚动位置全部消失 | 使用条件渲染（`{condition && <Component />}`）导致组件在视图切换时 unmount，React state 被销毁 | **有状态组件用 CSS `display:none` 控制显隐，不用条件渲染**：`<div className={cn('flex flex-col h-full', view !== 'chat' && 'hidden')}><Component /></div>`。组件始终挂载，state 保留 |
| FE-13 | Docker `npm install`/`npm ci` 失败报 `ERESOLVE could not resolve`，或 `tsc && vite build` 失败报 `TS6133: ... declared but never read`，本地开发却没问题 | **两个独立原因**：①Docker 使用 Node20/npm10，严格检查 peer deps（本地 npm11+ 默认宽松），`package-lock.json` 锁定了版本冲突的旧包；②`tsconfig.json` 开启了 `noUnusedLocals: true`/`noUnusedParameters: true`，条件渲染中被 `_` 包裹的参数、或某分支才用到的变量在 `tsc` 阶段报错，Docker build 中 `tsc && vite build` 因此失败不生成 dist | ①更新冲突的包到兼容版本（如 `npm install @tailwindcss/vite@4.2.2`），重新生成 lock file 提交；**不要**用 `npm install --legacy-peer-deps` 掩盖，要从根本修包版本；Dockerfile 用 `npm install` 不用 `npm ci`（后者对 peer deps 更严格）。②对条件分支下确实用不到的回调参数加 `_` 前缀（如 `_wid`、`_idx`），未使用的局部变量直接删除 |
| FE-14 | 页面变白 + 控制台 "React error #310" / "Hooks can only be called inside a function component" | conditional return（如认证守卫 `if (!user) { return <LoginPage/> }`）写在 `useState`/`useEffect`/`useCallback`/`useRef` 之前，某些渲染路径跳过了 hook，违反 Rules of Hooks | 将所有 hook 声明移到函数顶部；条件 return 必须在所有 hook 之后 |
| FE-15 | Write 覆写文件后，后续 StrReplace 将文件「回退」到旧结构混入新结构的混乱状态 | StrReplace 的 old_string 在旧文件和新文件内容中均可匹配（写入内容有重叠段落），StrReplace 执行了错误的替换，部分旧内容被重新混入 | **Write 覆写文件后必须先 Read 验证当前内容**，确认是新内容后再做后续 StrReplace；避免在 Write 之后立即基于「预期内容」写 StrReplace |

| FE-11 | 流式输出时用户向上翻页，界面立即跳回底部，无法查看历史消息 | `useEffect(() => { msgEndRef.current?.scrollIntoView() }, [messages])` 无条件在每次消息更新时滚动到底部，完全忽略用户是否正在向上翻页 | 用 `ref`（非 `state`）追踪用户是否在底部，仅在底部时自动滚动；向上翻时显示浮动「↓ 新消息」按钮；用户发送新消息时重置为底部锁定（详见下方完整代码）|

## FE-11 详细方案：流式输出智能滚动锁定

**适用场景**：所有含流式输出（SSE/流式响应）的聊天界面。

```typescript
// 1. 用 ref 追踪用户是否在底部（不触发重渲染）
const isAtBottomRef = useRef(true)
const scrollContainerRef = useRef<HTMLDivElement>(null)
const msgEndRef = useRef<HTMLDivElement>(null)
const [showScrollBtn, setShowScrollBtn] = useState(false)

// 2. 监听滚动事件，实时更新状态
const handleScroll = useCallback(() => {
  const el = scrollContainerRef.current
  if (!el) return
  const atBottom = el.scrollHeight - el.scrollTop - el.clientHeight < 80  // 80px 阈值容错
  isAtBottomRef.current = atBottom
  setShowScrollBtn(!atBottom)  // 控制「↓ 新消息」按钮显示
}, [])

// 3. 只在底部时才自动滚动
useEffect(() => {
  if (isAtBottomRef.current) {
    msgEndRef.current?.scrollIntoView({ behavior: 'smooth' })
  }
}, [messages])

// 4. 「回到底部」按钮：点击后重新锁定自动滚动
const scrollToBottom = useCallback(() => {
  msgEndRef.current?.scrollIntoView({ behavior: 'smooth' })
  isAtBottomRef.current = true
  setShowScrollBtn(false)
}, [])

// 5. 用户发送新消息时重置为底部锁定
const handleSend = () => {
  isAtBottomRef.current = true
  setShowScrollBtn(false)
  // ... 发送消息逻辑
}

// JSX 结构
// <div ref={scrollContainerRef} onScroll={handleScroll} style={{ overflowY: 'auto' }}>
//   {messages.map(...)}
//   <div ref={msgEndRef} />
// </div>
// {showScrollBtn && <button onClick={scrollToBottom}>↓ 新消息</button>}
```

**关键点**：
- 用 `ref` 而非 `state` 追踪滚动位置，避免不必要的重渲染
- 80px 的阈值容错，不需要精确到 0
- `scrollContainerRef` 挂在可滚动容器上，`msgEndRef` 挂在消息列表末尾的空 div

## 账号系统完整规范

### 注册流程（按顺序实现，不可跳过）

```
1. 手机号输入框（始终显示）
2. 发送验证码按钮（未填手机号时 disabled）
3. 验证码输入框（始终显示，发送前 disabled，发送后可输入）
4. 密码输入框
5. 用户名输入框
6. 注册按钮

API 调用顺序：
POST /api/auth/send-code  { phone }
POST /api/auth/register   { phone, code, password, username }
→ 注意参数顺序：(phone, code, password, username)
→ 成功后自动调用 login，获取 JWT Token
→ Token 存入 localStorage
```

### 登录流程

```
POST /api/auth/login  { phone, password }
→ 获取 { access_token, token_type, user }
→ token 存 localStorage
→ 检查用户状态 → 按状态跳转（HomeRedirect）
```

### Token 管理

```typescript
// api/auth.ts
const TOKEN_KEY = 'auth_token';

export const getToken = () => localStorage.getItem(TOKEN_KEY);
export const setToken = (token: string) => localStorage.setItem(TOKEN_KEY, token);
export const removeToken = () => localStorage.removeItem(TOKEN_KEY);

// axios 拦截器自动附加 token
axiosInstance.interceptors.request.use(config => {
  const token = getToken();
  if (token) {
    config.headers.Authorization = `Bearer ${token}`;
  }
  return config;
});
```

## FE-15：`overflow-hidden` 在 flex 容器中截断子组件，无法滚动

**触发场景**：父容器使用 `flex-1 overflow-hidden` 作为步骤内容区包装，子步骤组件（如文件上传页）内容超出容器高度时，超出部分被裁掉，内容不可滚动。

**错误现象**：文件列表增长到某个高度后，页面底部被截断，无任何滚动条出现。

**根因**：`overflow-hidden` 会裁剪超出内容，而不是触发滚动。在 flex 布局中，`flex-1` 子元素的高度由父容器决定，`overflow-hidden` 阻止了内容溢出，结果是超出部分直接不可见。

**正确做法**：
```tsx
// ❌ 错误：overflow-hidden 截断内容
<div className="flex-1 overflow-hidden">
  <Step1_FileUpload />
</div>

// ✅ 正确：overflow-y-auto 允许内容区滚动
<div className="flex-1 overflow-y-auto">
  <Step1_FileUpload />
</div>
```
**规律**：包裹步骤/内容区的 flex 容器，应用 `overflow-y-auto`（可滚动）而非 `overflow-hidden`（截断）。

## FE-16：`mt-auto` 按钮随文件列表增长被推出视口

**触发场景**：在 `flex flex-col h-full` 容器中，操作按钮使用 `mt-auto` 贴底显示。用户选择多个文件后，文件列表高度增长，按钮被推到容器底部之外，不可见。

**错误现象**：页面正常显示，但上传 / 确认按钮不见了；滚动到底部也找不到按钮。

**根因**：`h-full` 容器高度固定为视口高度，`mt-auto` 按钮贴底；文件列表（flex-grow 内容）扩张后超出容器，无 overflow 滚动，按钮在视口外。

**正确做法**（sticky footer 模式）：
```tsx
// 拆分为「可滚动内容区」+ 「固定按钮区」
<div className="flex flex-col h-full">
  {/* 上方：可滚动内容区 */}
  <div className="flex-1 overflow-y-auto px-6 py-4">
    <FileUploadZone ... />
    <PrivacyNotice />
  </div>

  {/* 下方：固定按钮区，始终可见 */}
  <div className="flex-shrink-0 px-6 py-3 border-t border-gray-100 bg-white">
    <button>上传</button>
  </div>
</div>
```
**规律**：有动态增长列表的页面，不要用 `mt-auto` 推按钮；改为 sticky footer（内容区 `flex-1 overflow-y-auto` + 按钮区 `flex-shrink-0`）。同时，动态列表本身应有 `max-h-[240px] overflow-y-auto` 约束，防止无限扩张。

## FE-17：wsId=null 导致发送/操作按钮静默失败（tashan-workbench）

**触发场景**：Tauri 桌面应用或 Web 应用中，组件在 `wsId && <Action>` 条件下渲染，初始化时 wsId 为 null，按钮点击无任何反应（handleSend 第一行 `if (!wsId) return`）。

**错误现象**：用户点击发送/上传按钮，页面无任何反应，无 loading 状态，无错误提示。

**根因**：
1. token resolver 优先读 localStorage，localStorage 里是旧 token
2. `getWorkspaces()` 用旧 token 请求返回 401，wsId 永远为 null
3. 按钮点击后 handler 第一行 `if (!wsId) return` 直接退出，没有任何提示

**正确做法**：
- 降低 localStorage token 的优先级（URL hash → 环境变量 → localStorage）
- `getWorkspaces()` 失败时清缓存并重试
- `wsId=null` 时必须给用户明确错误提示，而非静默失败
- Tauri sidecar 独立 token 场景：提供 demo/preview 模式不依赖 wsId
