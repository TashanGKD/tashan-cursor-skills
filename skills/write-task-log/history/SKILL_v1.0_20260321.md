# 任务日志填写技能

每次任务完成后，按以下步骤操作。**顺序不可颠倒，四步必须全部完成。**

> **设计原则（v2.0新增）**：默认模式是执行任务，不是捕捉认知。但每次任务完成后，自动提取认知价值，作为认知结构飞轮的燃料。郑总无需主动触发，这是系统自动完成的后台步骤。

---

## 步骤一：在 `任务日志.md` 追加日志条目

文件路径：`/Users/boyuan/aiwork/0310_huaxiang\_内部总控\任务日志.md`

### 格式模板

```markdown
---

## [TASK-YYYYMMDD-NN] 任务名称

**日期**：YYYY-MM-DD  
**执行者**：AI（Claude）  
**触发来源**：用户需求 / 联动规则XX / 内部计划  

### 完成的工作
（简明描述做了什么，关键决策是什么）

### 创建/修改的文件
- 新增：`路径/文件名`（说明用途）
- 修改：`路径/文件名`（说明改了什么）

### 关联关系
（本次任务与其他项目/文档的关系，使用 explains/implements/depends-on 等类型）

### 用户原始需求（逐条记录）
- 需求1：...

### 遗留事项
（未完成、需后续跟进的内容，无则写"无"）
```

### 规则
- ID 格式：`TASK-YYYYMMDD-NN`（当日第 NN 条，从 01 起）
- **只增不减**：绝对不修改历史条目，只在末尾追加
- Session 结束时，在最后一条日志末尾追加**需求汇总**（用户的明确需求 + 隐含设计倾向 + 对未来方向的暗示）

文件路径：`/Users/boyuan/aiwork/0310_huaxiang\_内部总控\项目排期表.md`

```
检查该任务是否已在排期表中？
  ├── 已存在 → 更新状态字段（🔲待开始 → 🟡进行中 → ✅完成）
```

## 步骤四：认知提取（v2.0新增，任务完成后自动运行）

**触发**：每次任务完成后自动执行，不需要郑总主动触发。

**执行方法：**

```
回顾本次任务的完整对话，扫描三类信号：

类型1：郑总在任务中表达了判断/原则/洞见
  示例：「我觉得这个设计原则是...」「这背后的本质是...」
  → 如果值得长期保留：视为 L2 碎片，进入 cognitive-capture-fragment 流程

类型2：任务过程暴露了 Skill 或规范的漏洞
  示例：AI做错了什么、规范不覆盖某种情况、Skill 触发错误
  → 记录到 issue-tracker 或认知结构维护协议待处理项

类型3：有内容应该整合进认知结构 L1 文档
  示例：本次任务产生了系统性的新知识，对应某个 L1 文档
  → 标记为「待整合」，加入 L2 碎片整合索引
```

**执行要求（F7修复，2026-03-19）：**

⚠️ **「写描述」≠「实际捕捉」**。以下操作必须真实执行，不能只写在任务日志里：

```
发现类型1（碎片）：
  → 真正在 L2 碎片文件末尾追加碎片内容（Write 工具）
  → 真正在碎片整合索引.md 追加新行（Write 工具）
  → 不是写「已加入L2」的文字描述，而是真的写进去

发现类型2（漏洞）：
  → 真正在 L3/待完成总清单.md 或维护协议追加条目（Write 工具）

发现类型3（L1整合）：
  → 真正在碎片整合索引.md 标注「🔲待整合」（Write 工具）
```

**输出汇报（在任务日志追加，最多3条）：**

```markdown
### 认知产出（自动提取，已实际写入）
- [碎片F-013] 「...」→ 已写入 L2/产品思考碎片/（碎片整合索引已更新）
- [漏洞→待处理] Skill X 在场景Y下无法处理 → 已加入待完成总清单
- [整合标记] L1[A] 第X章建议补充「...」→ 已标注为🔲待整合
```

**原则：**
- 无价值内容 → 静默跳过，不汇报，不问郑总
- 有价值内容 → 必须先**实际执行写入**，再汇报「已写入」
- 认知提取是后台飞轮，不阻断工作流

## 快速检查清单

完成任务后，逐项确认：

- [ ] `任务日志.md` 末尾已追加新条目（格式正确，ID 唯一）
- [ ] `项目排期表.md` 状态已更新
- [ ] 跨项目联动已评估（有触发则处理，无则记录"已评估，无需联动"）
- [ ] 步骤四认知提取已执行（有价值内容则追加「认知产出」，无则静默）
- [ ] 若本次 Session 结束，在日志末尾追加需求汇总
