# Skill 闭环经验捕捉（skill-capture-closure）

> 把刚跑通（或踩坑）的经验沉淀为 Skill，是 Skill 体系自我进化的核心机制。
> 类比：认知结构的 cognitive-capture-fragment，但作用对象是 AI 行为规范而非个人思考。

---

## 激活后立即执行（顺序不可跳过）

```
Step 1  读取索引，了解现有 Skill 全貌
        Read: .cursor/skills/skill-index/SKILL-INDEX.md
        → 获取所有现有 Skill 的目录名、状态、版本号

Step 2  理解本次经验
        询问用户（如果上下文不明确）：
        「这次跑通/踩坑的是什么任务？核心发现是什么？」
        → 要求得到：任务类型 + 经验核心（一句话）+ 是成功经验还是踩坑记录

Step 3  [关卡A] 判断归属：新建 Skill，还是完善已有 Skill？
        → 遍历索引，语义匹配：是否有现有 Skill 覆盖了相同的任务类型？
        
        【情况A：高度重合（语义相似度>70%）】
        → 告知用户：「此经验与已有 Skill [目录名] 高度相关。
          建议：完善该 Skill，而非新建。确认？」
          → 用户确认 → 跳到 Step 6（更新已有 Skill）
          → 用户说"不，这是新的" → 继续 Step 4
        
        【情况B：有相关但不重合（补充场景/边界条件）】
        → 告知用户：「此经验可作为 [目录名] Skill 的补充场景/注意事项。
          建议：追加到该 Skill 的对应章节，而非新建。确认？」
          → 用户确认 → 跳到 Step 6（追加到已有 Skill）
          → 用户说"不，新建" → 继续 Step 4
        
        【情况C：无相关现有 Skill】
        → 继续 Step 4（新建 Skill）

Step 4  [新建流程] 设计新 Skill 的基本属性
        推断并向用户确认：
        
        「准备新建 Skill：
          目录名：[kebab-case命名，最多6段，含语义]
          类别：[角色类 | 认知系统类 | 科研画像类 | 系统治理类 | 写作类]
          触发词：[3-5个关键词]
          核心能力（一句话）：[...]
          [确认新建] [调整后新建]」

Step 5  [新建流程] 生成并写入 SKILL.md
        a. 创建目录：.cursor/skills/[目录名]/
        b. 写入 SKILL.md，包含：
           - YAML frontmatter（name + description含触发词）
           - 激活后立即执行的步骤（顺序明确）
           - 注意事项（来自本次踩坑或经验）
           - 变更记录章节（追加在文件底部）：
             格式见下方「变更记录格式」
        c. 如有辅助文件需要（示例/参考文档），一并创建
        → 跳到 Step 7

Step 6  [更新流程] 受控更新已有 Skill
        必须先执行 skill-rule-修改规范 的「修改前三问」：
        1. 根因：本次触发更新的真实事件是什么？
        2. 影响：哪些已有执行路径会改变？
        3. 验证：如何验证修改有效且无副作用？
        
        三问回答完毕后：
        a. 备份：Copy 原文件到 .cursor/skills/[目录名]/history/SKILL_v旧版_YYYYMMDD.md
        b. 执行修改（最小化原则：只改需要改的）
        c. 在 SKILL.md 底部追加变更记录（格式见下方）

Step 7  更新 Skill 总索引
        Read: .cursor/skills/skill-index/SKILL-INDEX.md
        
        【版本号规则（内联，执行时无需翻其他文件）】
        - 核心逻辑/步骤顺序/新增章节变动 → 第二位 +1（v1.0 → v1.1，v1.1 → v1.2）
        - 注意事项/触发词/description/文字修正 → 第三位 +0.1（v1.0 → v1.0.1）
        - 全新创建 → v1.0
        
        【新建 Skill】→ 在对应类别表格追加一行：
        | [目录名] | v1.0 | 🔵 待验证 | [今日日期] | [触发词] | [核心能力] |
        
        【更新已有 Skill（经验完善）】→ 更新对应行：
        - 版本号按上方规则递增
        - 状态改为 ✅ 已验证（如果本次是成功闭环经验）
        - 最近更新日期 → 今日
        
        【更新已有 Skill（踩坑修复）】→ 更新对应行：
        - 版本号按上方规则递增
        - 若修复前发现问题，先改为 ⚠️ 有已知问题，修复后改回 ✅ 已验证
        - 最近更新日期 → 今日
        
        同时在索引底部「变更记录」追加一行：
        [日期] | [Skill目录名] | v旧→v新 / 新建v1.0 | [修改一句话摘要]

Step 8  收尾反馈
        「✅ 经验已沉淀：
          [新建/更新] Skill：[目录名] v[版本号]
          索引已同步更新
          
          Skill 体系健康度：共 N 个 Skill，✅ X 个已验证，🔵 Y 个待验证
          [查看总索引] [继续当前任务]」
```

## 变更记录格式

每次新建或更新 Skill 时，在 SKILL.md 底部追加。**本格式与 `skill-rule-修改规范` 统一，是其超集**：

```markdown
---

## 变更记录

### v[版本号] — YYYY-MM-DD — [变更标题（一句话）]

**根因**：[触发本次修改/创建的真实事件——是什么任务/闭环/踩坑导致了这次沉淀]

**经验核心**：[这次学到了什么 / 踩了什么坑 / 修复了什么问题]

**修改内容**：
- 新增：[章节/条目] → [一句话说明]
- 修改：[章节/条目] → [从什么改成了什么]
- 删除：[章节/条目] → [为什么删除]

**验证结果**：
- 正向验证：[场景] → [结果：通过 / 待验证]
- 负向验证：[场景] → [结果：通过 / 待验证]

**验证状态**：🔵 待验证 / ✅ 已在真实场景跑通
```

> **说明**：新建 Skill 时「验证结果」填「首次创建，验证将在第一次实际使用时执行」；更新 Skill 时必须填写正向/负向验证描述。

---

## 判断「值得沉淀」的标准

并非每次任务完成都要沉淀 Skill。以下情况**值得沉淀**：

| 情况 | 说明 | 沉淀方式 |
|---|---|---|
| 跑通了一个之前不确定的闭环 | 比如第一次成功完成某类复杂任务 | 新建 or 更新 Skill，状态→✅ 已验证 |
| 踩了一个坑并找到了解决方案 | 有「如果早知道就好了」的感觉 | 更新对应 Skill 的「注意事项」 |
| 发现某个 Skill 的步骤顺序有问题 | 执行时发现某步骤应该更早/晚 | 更新 Skill 步骤，追加变更记录 |
| 发现某个 Skill 的触发词不准 | 该触发的时候没触发，或触发了不该触发的 | 更新 description 字段，版本+0.1 |
| 发现两个 Skill 的行为有冲突 | AI 不知道该执行哪个 | 在两个 Skill 中互相标注优先级 |

**不值得沉淀**：重复执行了已有闭环，没有新发现。

---

## 注意事项

- **最小化原则**：更新已有 Skill 时，只改本次经验相关的部分，不顺手重构整个文件
- **版本号规则**：大改（步骤顺序/核心逻辑变动）→ v1.x+1；小改（注意事项/触发词）→ v1.x+0.1
- **索引必须同步**：只要动了 Skill，索引就要同步更新；不允许索引与实际状态不符
- **不替代 skill-rule-修改规范**：本 Skill 调用 skill-rule-修改规范 的「修改前三问」，不绕过它
