# Skill 闭环经验捕捉（skill-capture-closure）

> 把刚跑通（或踩坑）的经验沉淀为 Skill，是 Skill 体系自我进化的核心机制。
> 类比：认知结构的 cognitive-capture-fragment，但作用对象是 AI 行为规范而非个人思考。

---

## 激活后立即执行（顺序不可跳过）

```
Step 1  读取索引，了解现有 Skill 全貌
        Read: .cursor/skills/skill-index/SKILL-INDEX.md
        → 获取所有现有 Skill 的目录名、状态、版本号

Step 2  理解本次经验
        询问用户（如果上下文不明确）：
        「这次跑通/踩坑的是什么任务？核心发现是什么？」
        → 要求得到：任务类型 + 经验核心（一句话）+ 是成功经验还是踩坑记录

Step 2.1  【结构性根因诊断】——表层执行问题 vs 设计层问题

        ⚠️ 这是反思质量的关键门槛：AI 反思时容易只看到「我疏忽了」，
        但真正的根因可能是「规范本身设计有误，AI 即使尽力也无法可靠执行」。

        必须回答以下诊断问题序列：

        ────────────────────────────────────────
        问题A：这次 AI 是否知道相关规范的存在？

        → 【不知道】→ 根因是【连接缺口】：
          规范/文档没有通过 D0 / alwaysApply Rule / Step 1 进入 AI 的上下文。
          AI 无从遵守一个不知道存在的规范。
          → 产物：修复连接机制（补 D0 行 / 新建 Rule / 调整引用位置）
          → 继续 Step 2.5（类型R 或 类型D）

        ────────────────────────────────────────
        → 【知道规范，但没执行】→ 继续问题B：

        问题B：规范是通过什么机制传递的？

        → 只在某个被触发的 Skill 里（Skill 这次没被触发）：
          根因：【类型错误】该规范的普遍性要求它是 alwaysApply Rule，
          而不是只有一个 Skill 被调用时才生效的注意事项。
          → 产物：将该规范从 Skill 注意事项提升为 Rule
          → 继续 Step 2.5（类型R）

        → 规范确实在 Rule/D0 中，但表述模糊/有歧义：
          根因：【规范质量问题】规范写得让 AI 有多种理解，或覆盖场景不明确。
          → 产物：重写规范（修订规范文档本身，使其更明确）
          → 继续 Step 2.5（类型D 或 类型S）

        → 规范表述是「建议/应该」而非「必须/禁止」：
          根因：【执行力度不足】AI 在压力下会跳过「建议」，不会跳过「禁止」。
          → 产物：将「建议」改为「禁止不执行」或添加硬性检查门
          → 继续 Step 2.5（类型R 或 类型S）

        → 规范在正确位置，表述清晰，AI 确实就是疏忽了：
          根因确实是【执行层问题】——AI 这次没有尽职执行
          → 正常继续 Step 2.5（加注意事项/提醒即可）

        ────────────────────────────────────────
        输出诊断结论（一句话）：
        「本次根因：[连接缺口/类型错误/规范质量/执行力度/执行层疏忽]——[一句话说明]」
        → 这句话必须写入本次经验的变更记录「经验核心」字段

Step 2.5  【四型前置分类】本次经验应沉淀为哪种类型的产物？

        基于 Skill体系设计原则_v1.0.md §2.5 三型决策树判断：

        【类型R：应建/改 Rule】
        判断标准：这个洞见是「所有任务都必须遵守的普遍约束」，轻量，违反有明确负面后果
        例子：「每次修改文件前必须备份」「任何时候不能硬编码 API Key」
        → 分支R1：若现有 Rule 可以覆盖 → 走 Step R1（更新已有 Rule）
        → 分支R2：无现有 Rule → 走 Step R2（新建 Rule）

        【类型A：应建/改 Agent】
        判断标准：这个任务需要独立视角隔离（审核/评测）或并行批量执行
        例子：「需要一个独立的验证者视角」「批量处理多份文档」
        → 分支A1：若现有 Agent 可以覆盖 → 走 Step A1（更新已有 Agent）
        → 分支A2：无现有 Agent → 走 Step A2（新建 Agent）

        【类型D：应更新规范/参考文档】
        判断标准：洞见是某份参考文档（非 .cursor/ 体系组件）的内容有误或需要补充
        覆盖：开发规范手册/技术架构文档/测试规范/任何项目或总规范文档
        → 走 Step D（更新规范文档）

        【类型S：应建/改 Skill（原有流程）】
        判断标准：上述三类都不符合，即「执行某类任务的步骤化流程」
        → 继续 Step 3（原有 Skill 归属判断）

        ⚠️ 若类型不确定 → 问用户：「这个经验是 ① 普遍约束（Rule）② 需要独立视角（Agent）③ 规范文档内容有误 ④ 任务执行步骤（Skill），哪个更准确？」

Step R1  【更新已有 Rule】
        Read: .cursor/rules/[对应rule文件]（先了解现状）
        执行 skill-rule-修改规范 完整流程：
        → 修改前三问（根因/影响/验证）
        → 备份到 history/
        → 最小化修改
        → 追加变更记录
        → 更新 SKILL-INDEX（Rule变更行）
        → 跳到 Step 8（收尾反馈）

Step R2  【新建 Rule】
        基于经验设计 Rule：
        - name + description（含触发条件）
        - alwaysApply: true/false 判断
        - 核心约束内容（< 300词，只写约束，不写流程）
        Write: .cursor/rules/[rule-name].mdc
        备份不需要（新建）；追加变更记录到 Rule 文件底部
        同步检查：新 Rule 是否需要配套 Skill？（skill-rule-修改规范 第4条）
        → 跳到 Step 8（收尾反馈）

Step A1  【更新已有 Agent】
        Read: .cursor/agents/[对应agent文件]（先了解现状）
        执行 skill-rule-修改规范 完整流程（Agent 文件适用相同规范）：
        → 修改前三问（根因/影响/验证）
        → 备份到 .cursor/agents/history/
        → 最小化修改
        → 追加变更记录到 Agent 文件
        → 更新 SKILL-INDEX Agents 表（若有）
        → 跳到 Step 8

Step A2  【新建 Agent】
        需要走 skill-designer Level 2 流程（含关卡A + 关卡C）：
        → 告知用户：「这需要新建一个独立子智能体，建议说「新建Agent」触发 skill-designer」
        → 或由本次对话上下文直接引导 skill-designer 设计
        → 跳到 Step 8

Step D  【更新规范/参考文档】
        识别目标文档：
        - 开发规范类：_内部总控/开发规范/*.md
        - 项目规范类：项目群/[项目]/测试工程师/测试规范.md 等
        - 其他参考文档

        执行 project-doc-versioning-guard 四步留痕：
        1. 读取文件头部，获取当前版本号
        2. cp [文件] → [文件所在目录]/history/[文件名]_v[版本]_YYYYMMDD.md
        3. 执行修改（最小化，只改本次经验相关的部分）
        4. 在文档末尾追加变更记录：
           ### v[新版本] — YYYY-MM-DD — [本次经验触发的修改]
           **修改内容**：[一句话说明]
           **触发事件**：[本次踩坑/发现描述]
        → 跳到 Step 8

Step 3  [关卡A] 判断归属：新建 Skill，还是完善已有 Skill？
        → 遍历索引，语义匹配：是否有现有 Skill 覆盖了相同的任务类型？
        
        【情况A：高度重合（语义相似度>70%）】
        → 告知用户：「此经验与已有 Skill [目录名] 高度相关。
          建议：完善该 Skill，而非新建。确认？」
          → 用户确认 → 跳到 Step 6（更新已有 Skill）
          → 用户说"不，这是新的" → 继续 Step 4
        
        【情况B：有相关但不重合（补充场景/边界条件）】
        → 告知用户：「此经验可作为 [目录名] Skill 的补充场景/注意事项。
          建议：追加到该 Skill 的对应章节，而非新建。确认？」
          → 用户确认 → 跳到 Step 6（追加到已有 Skill）
          → 用户说"不，新建" → 继续 Step 4
        
        【情况C：无相关现有 Skill】
        → 继续 Step 4（新建 Skill）

Step 4  [新建流程] 设计新 Skill 的基本属性
        推断并向用户确认：
        
        「准备新建 Skill：
          目录名：[kebab-case命名，最多6段，含语义]
          类别：[角色类 | 认知系统类 | 科研画像类 | 系统治理类 | 写作类]
          触发词：[3-5个关键词]
          核心能力（一句话）：[...]
          [确认新建] [调整后新建]」

Step 5  [新建流程] 生成并写入 SKILL.md
        a. 创建目录：.cursor/skills/[目录名]/
        b. 写入 SKILL.md，包含：
           - YAML frontmatter（name + description含触发词）
           - 激活后立即执行的步骤（顺序明确）
           - 注意事项（来自本次踩坑或经验）
           - 变更记录章节（追加在文件底部）：
             格式见下方「变更记录格式」
        c. 如有辅助文件需要（示例/参考文档），一并创建
        → 跳到 Step 7

Step 6  [更新流程] 受控更新已有 Skill
        必须先执行 skill-rule-修改规范 的「修改前三问」：
        1. 根因：本次触发更新的真实事件是什么？
        2. 影响：哪些已有执行路径会改变？
        3. 验证：如何验证修改有效且无副作用？
        
        三问回答完毕后：
        a. 备份：Copy 原文件到 .cursor/skills/[目录名]/history/SKILL_v旧版_YYYYMMDD.md
        b. 执行修改（最小化原则：只改需要改的）
        c. 在 SKILL.md 底部追加变更记录（格式见下方）

Step 7  更新 Skill 总索引
        Read: .cursor/skills/skill-index/SKILL-INDEX.md
        
        【版本号规则（内联，执行时无需翻其他文件）】
        - 核心逻辑/步骤顺序/新增章节变动 → 第二位 +1（v1.0 → v1.1，v1.1 → v1.2）
        - 注意事项/触发词/description/文字修正 → 第三位 +0.1（v1.0 → v1.0.1）
        - 全新创建 → v1.0
        
        【新建 Skill】→ 在对应类别表格追加一行：
        | [目录名] | v1.0 | 🔵 待验证 | [今日日期] | [触发词] | [核心能力] |
        
        【更新已有 Skill（经验完善）】→ 更新对应行：
        - 版本号按上方规则递增
        - 状态改为 ✅ 已验证（如果本次是成功闭环经验）
        - 最近更新日期 → 今日
        
        【更新已有 Skill（踩坑修复）】→ 更新对应行：
        - 版本号按上方规则递增
        - 若修复前发现问题，先改为 ⚠️ 有已知问题，修复后改回 ✅ 已验证
        - 最近更新日期 → 今日
        
        同时在索引底部「变更记录」追加一行：
        [日期] | [Skill目录名] | v旧→v新 / 新建v1.0 | [修改一句话摘要]

Step 7.5  role-menu 同步检查（SD1 修复）
        ⚠️ 新建 Skill 时必须执行本步骤；更新已有 Skill 时可跳过。
        
        判断标准：新 Skill 是否应该出现在 role-menu.mdc「可用角色一览」表中？
        
        应加入 role-menu 的 Skill（满足任一条件）：
        □ 用户会直接说某句话来触发它（即它是一个用户直接触发的入口）
        □ 它是角色类 Skill（role-* 目录名）
        □ 它有新的触发词，与现有 role-menu 中的 Skill 不重叠
        
        不需要加入 role-menu 的 Skill：
        □ 由其他 Skill/Agent 调用的能力层 Skill（如 ai-image-generator，在 document-pipeline 内调用）
        □ 认知系统内部流转的子 Skill（如 cognitive-verifier，由 update-knowledge 调用）
        
        IF 应加入 role-menu：
          → 告知用户：「新 Skill [目录名] 需要加入 role-menu.mdc。
             建议触发词：[description 中的触发词]
             建议添加到「可用角色一览」和「使用方式」两处。
             [立即更新 role-menu] [稍后手动更新]」
          → 用户选「立即更新」→ 按 skill-rule-修改规范 备份 role-menu.mdc 后追加行
          → 用户选「稍后」→ 追加到 PENDING-SKILLS.md 待处理
        IF 不需要加入：
          → 静默跳过

Step 8  收尾反馈
        「✅ 经验已沉淀：
          [新建/更新] Skill：[目录名] v[版本号]
          索引已同步更新
          
          Skill 体系健康度：共 N 个 Skill，✅ X 个已验证，🔵 Y 个待验证
          [查看总索引] [继续当前任务]」
```

## 变更记录格式

每次新建或更新 Skill 时，在 SKILL.md 底部追加。**本格式与 `skill-rule-修改规范` 统一，是其超集**：

```markdown
---

## 变更记录

### v[版本号] — YYYY-MM-DD — [变更标题（一句话）]

**根因**：[触发本次修改/创建的真实事件——是什么任务/闭环/踩坑导致了这次沉淀]

**经验核心**：[这次学到了什么 / 踩了什么坑 / 修复了什么问题]

**修改内容**：
- 新增：[章节/条目] → [一句话说明]
- 修改：[章节/条目] → [从什么改成了什么]
- 删除：[章节/条目] → [为什么删除]

**验证结果**：
- 正向验证：[场景] → [结果：通过 / 待验证]
- 负向验证：[场景] → [结果：通过 / 待验证]

**验证状态**：🔵 待验证 / ✅ 已在真实场景跑通
```

> **说明**：新建 Skill 时「验证结果」填「首次创建，验证将在第一次实际使用时执行」；更新 Skill 时必须填写正向/负向验证描述。

## 判断「值得沉淀」的标准

并非每次任务完成都要沉淀 Skill。以下情况**值得沉淀**：

| 情况 | 说明 | 沉淀方式 |
|---|---|---|
| 跑通了一个之前不确定的闭环 | 比如第一次成功完成某类复杂任务 | 新建 or 更新 Skill，状态→✅ 已验证 |
| 踩了一个坑并找到了解决方案 | 有「如果早知道就好了」的感觉 | 更新对应 Skill 的「注意事项」 |
| 发现某个 Skill 的步骤顺序有问题 | 执行时发现某步骤应该更早/晚 | 更新 Skill 步骤，追加变更记录 |
| 发现某个 Skill 的触发词不准 | 该触发的时候没触发，或触发了不该触发的 | 更新 description 字段，版本+0.1 |
| 发现两个 Skill 的行为有冲突 | AI 不知道该执行哪个 | 在两个 Skill 中互相标注优先级 |

**不值得沉淀**：重复执行了已有闭环，没有新发现。

---

## 注意事项

- **最小化原则**：更新已有 Skill 时，只改本次经验相关的部分，不顺手重构整个文件
- **版本号规则**：大改（步骤顺序/核心逻辑变动）→ v1.x+1；小改（注意事项/触发词）→ v1.x+0.1
- **索引必须同步**：只要动了 Skill，索引就要同步更新；不允许索引与实际状态不符
- **不替代 skill-rule-修改规范**：本 Skill 调用 skill-rule-修改规范 的「修改前三问」，不绕过它
