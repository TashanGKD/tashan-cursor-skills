## 踩坑速查表

| # | 症状 | 根因 | 修复方案 |
|---|---|---|---|
| BE-01 | 所有 API 请求静默失败（无报错，但无响应） | `async with` 用在同步的 postgres_client 上 | 改用 `with get_db_session() as session:` 同步模式 |
| BE-02 | 注册接口返回 422 | register 参数顺序错误 | 按 `(phone, code, password, username)` 顺序 |
| BE-03 | JSONB 字段写入后为空 / None | `:state::jsonb` 语法中 `::jsonb` 被 SQLAlchemy 解析为参数前缀，`state` 参数被静默丢弃 | **改为 `CAST(:state AS jsonb)`**，写完任何 JSONB 写入函数立即测试 |
| BE-04 | `KeyError: 'id'` 在业务代码 | JWT payload 字段不一致：TopicLab 用 `sub`，业务代码期待 `id` | 实现 `_normalize_user()` 统一补全 `id` 和 `sub` |
| BE-05 | 跨项目 schema 引用报错 | `auth.py` 引用 `topiclab.users`，但实际表在 `public` schema | 改为 `public.users`（跨项目统一） |
| BE-06 | SSE 连接成功但客户端永远转圈 | DB 写入失败被静默忽略，SSE generator 没有全局 try/except | **所有 SSE generator 必须有全局 try/except**，异常时输出 error event |
| BE-07 | 函数名说用 LLM，但没有 AI 效果 | `_plan_with_llm()` 内部是硬编码规则，命名欺骗 | 按诚实命名原则重命名为 `_plan_with_rules()`，加 TODO 注释 |
| BE-08 | uvicorn 不断重载，服务不稳定 | 临时调试脚本（如 check_db.py）放在 backend/ 目录下，watchfiles 触发重载 | 临时脚本放在项目根目录或 `/tmp`，不放 uvicorn 监控的代码目录 |
| BE-09 | 多用户数据泄露（用户A看到用户B数据） | SQL 查询没有加 `WHERE user_id = :user_id` 过滤 | 所有涉及用户数据的查询，必须加 user_id 过滤 |
| BE-10 | 环境变量缺失时系统启动成功但运行时崩溃 | 环境变量在第一次用到时才读取，没有启动时校验 | 在 `config/settings.py` 中，所有必须的环境变量用 `os.environ` 读取（无默认值），缺失则启动时 raise |
| BE-11 | 登录/注册返回 500，前端报 `Unexpected token 'I'` | `sessionmaker(autocommit=True)` 在 SQLAlchemy 2.x 中已移除，抛出 `ArgumentError` | 改为 `sessionmaker(autocommit=False)`，context manager 里显式 `commit()`/`rollback()` |
| BE-12 | Docker build 时 pip 报 `--index-url option requires 1 argument` | Dockerfile 中 `ARG PIP_INDEX_URL` 在 `FROM` 之前声明，`FROM` 之后变量值变为空 | **ARG 必须在 FROM 之后重新声明**才能在 RUN 中使用 |
| BE-13 | FastAPI `StreamingResponse` generator 末尾的副作用代码（文件写入、日志等）不执行 | Python 同步 generator 在 `StopIteration` 后，`finally` 块中 `yield` 之后的代码依赖 GC 时机，`StreamingResponse` 的迭代器消费完后不保证立即执行 | **副作用代码必须放在最后一次 `yield` 之前**：`_save_side_effect(); yield "data: [DONE]\\n\\n"`，不要放在 `finally` 块内 `yield` 的后面 |
| BE-14 | passlib hash_password 失败（AttributeError/__about__ 或 ValueError/72bytes）| passlib 1.7.x 与 bcrypt 4.1+ API 不兼容（detect_wrap_bug 使用了已移除的内部属性）| requirements.txt 锁定 `bcrypt==4.0.1`，这是与 passlib 1.7.x 兼容的最后版本 |
| BE-15 | 跨项目复用后端文件时，容器启动后 502 + import 报错（如 ImportError: cannot import name auth_bridge）| 直接 Copy-Item 整个文件覆盖，导致引入了目标项目不存在的 import（源项目特有依赖）| **不整体覆盖**：先 git diff 确认改动范围，只做最小外科手术式追加。如需整体复用，必须逐行检查 import，删除不存在的依赖 |
| BE-16 | 登录后 `workspace_id` 为 null，误触发不该触发的 UI（如创作者激活弹窗） | nginx 把 `/api/auth/` 统一路由到外部服务，login/me 绕过本产品 Python 的 `workspace_id` 附加逻辑 | 只精确路由无本域特有字段的端点（如 `location = /api/auth/send-code`）；login/me 保留本服务；改路由前输出 API 契约检查表（`role-后端开发` Step 0.2） |
| BE-17 | 生产未配置 `AUTH_SERVICE_BASE_URL` 等 env 时认证代理直接 503 | 用 httpx 做应用层「代理转发」属间接复用，增加 env、Docker 网络、超时等依赖点 | 共用账号/认证走 nginx 层 `proxy_pass`；Python 只处理本域逻辑（`workspace_id` 附加、creator-activate 等）；参考 tashan-openbrain `deploy.yml` 的 nginx auth 路由段 |

## PostgreSQL 连接规范

### 正确用法

```python
# postgres_client.py（来自 Tashan-TopicLab）
from contextlib import contextmanager
from sqlalchemy import create_engine
from sqlalchemy.orm import sessionmaker

engine = create_engine(DATABASE_URL, pool_pre_ping=True)
SessionLocal = sessionmaker(bind=engine)

@contextmanager
def get_db_session():
    session = SessionLocal()
    try:
        yield session
        session.commit()
    except Exception:
        session.rollback()
        raise
    finally:
        session.close()

# 使用方式（同步）
def get_user(user_id: int):
    with get_db_session() as session:
        result = session.execute(
            text("SELECT * FROM public.users WHERE id = :id"),
            {"id": user_id}
        ).fetchone()
        return result
```

### 常见 SQL 注意事项

```python
# JSONB 写入（必须用 CAST 语法）
session.execute(
    text("""
        INSERT INTO t (id, data) 
        VALUES (:id, CAST(:data AS jsonb))
        ON CONFLICT (id) DO UPDATE SET data = CAST(:data AS jsonb)
    """),
    {"id": 1, "data": json.dumps(my_dict)}
)

# 数组写入
session.execute(
    text("INSERT INTO t (tags) VALUES (CAST(:tags AS text[]))"),
    {"tags": "{tag1,tag2}"}
)

# 时间戳（用 UTC）
session.execute(
    text("INSERT INTO t (created_at) VALUES (NOW() AT TIME ZONE 'UTC')"),
    {}
)
```

## BE-16：nginx 路由过度导致 OpenBrain 特有字段（workspace_id）丢失

**触发场景**：把 `/api/auth/` 统一路由到外部服务（如 topiclab），外部服务不返回本产品特有字段（如 workspace_id），导致前端消费方字段为 null。

**错误现象**：所有用户登录后 `workspace_id` 为 null，触发不该触发的 UI 流程（如创作者激活弹窗）。

**根因**：nginx 路由变更（`location /api/auth/`）把 login 和 /me 也路由出去，绕过了 OpenBrain Python 的 workspace_id 附加逻辑。改路由时没有检查「下游消费方依赖什么字段」（违反 L1.5 P17）。

**正确做法**：
- 只路由「无本产品特有数据」的端点（send-code / register）用 nginx 精确匹配 `location = /api/auth/send-code`
- 需要附加业务数据的端点（login / me）保留在本产品 Python 处理
- 改路由前必须先输出 API 契约检查表（见 role-后端开发 Step 0.2）

**验证**：修复后 login 和 /me 重新返回 workspace_id，smoke test 通过。


## BE-17：Python httpx 代理 ≠「直接用已有模块」

**触发场景**：项目共用另一个服务的账号系统，AI 写了 Python httpx 代理转发请求。

**错误现象**：Python 代理需要额外 env var（AUTH_SERVICE_BASE_URL），在生产未配置时直接 503，而不是透明复用。

**根因**：把「复用」理解成「应用层代理」，实际上这是「间接复用」——增加了依赖点（env var、Docker 网络、httpx 超时）。真正的「直接用」是在基础设施层（nginx 路由），把请求直接转发，Python 完全不介入。

**正确做法**：
- 共用账号/认证服务 → nginx 层路由（`location = /api/auth/send-code` proxy_pass 到目标服务端口）
- Python 只处理本产品特有的业务逻辑（workspace_id 附加、creator-activate 等）
- 参考：tashan-openbrain `deploy.yml` 的 nginx auth 路由段

## BE-18：Tauri 桌面应用弹窗/模态框永远不显示

**触发场景**：Tauri 桌面应用使用 sidecar 二进制作为后端，前端存在依赖 wsId 非空才渲染的弹窗组件（如 BrainConfirmModal）。

**错误现象**：弹窗组件（BrainConfirmModal 等）在 Tauri 应用中永远不显示，即使触发条件看起来已满足。

**根因**：
- Tauri sidecar 二进制有自己独立生成的认证 token，与前端 `.env` 中写死的开发 token 不匹配
- 导致 `getWorkspaces()` 等初始化请求失败，`wsId` 始终为 null
- 含 `wsId && <Modal>` 条件的弹窗渲染判断永远为 false

**正确做法**：
- Tauri 应用的弹窗/模态框渲染不得硬性依赖 `wsId` 非空
- 提供不依赖后端连接的 demo/preview 模式，或在初始化失败时提供 fallback wsId
- 调试时优先检查 sidecar token 是否与前端 .env token 一致

**来源**：tashan-workbench 项目，2026-03-21

## BE-19：前端读取服务端字段显示 undefined（camelCase vs snake_case 数据契约不一致）

**触发场景**：前后端使用不同语言/运行时（如后端 Bun TypeScript，前端 React TypeScript），API 响应字段命名风格未在技术架构文档中明确约定。

**错误现象**：前端组件无法正确读取服务端返回的数据，字段显示 undefined。

**根因**：
- 服务端（Bun TypeScript）自然返回 camelCase：`questionDraft`, `canRevoke`, `createdAt`
- 前端 TypeScript 类型定义使用 snake_case：`question_draft`, `can_revoke`, `created_at`
- 数据契约不一致，前端解构/访问时字段名不匹配，得到 undefined
- verifier 独立测试才发现，集成测试未专门验证响应字段名

**正确做法**：
- 在技术架构文档中明确约定 API 命名风格（camelCase 或 snake_case），全栈统一
- 前后端类型定义（TypeScript interface / Pydantic model）保持一致
- 集成测试中专门验证响应字段名，不只验证字段是否存在

**来源**：tashan-workbench 项目，2026-03-21

## BE-20：JWT payload 不含 workspace_id 导致 workspace 隔离失效

**触发场景**：FastAPI 项目中，`inject_workspace` 依赖从 JWT token 获取 `workspace_id`，但 `create_access_token` 生成 JWT 时没有包含该字段。

**错误现象**：普通用户调用任何需要 workspace 隔离的接口（如 `/doc-registry`、`/twin/upload`），返回的却是 admin workspace 的内容（如 20 篇 admin 认知文档），而非自己的空 workspace。

**根因**：
- `create_access_token` payload 只包含 `{"sub": user_id, "id": user_id, "role": role, "exp": ...}`
- `inject_workspace` 中 `user.get("workspace_id")` 返回 None
- 因此 fallback 到 `settings.ADMIN_WORKSPACE_ID`，所有用户都看到 admin workspace

**正确做法**：
```python
# 方案 A（推荐）：inject_workspace 中从 DB 查询用户 workspace（不依赖 JWT）
async def inject_workspace(...):
    ...
    default_ws = user.get("workspace_id") or ""
    if not default_ws and is_db_available():
        from core.auth import get_profile
        with get_db() as session:
            profile = get_profile(session, user["id"])
            if profile:
                default_ws = profile.get("workspace_id", "")
    ...

# 方案 B：在 JWT 中包含 workspace_id（切换分身时需重新颁发 token）
payload = {"sub": str(user_id), "id": user_id, "role": role,
           "workspace_id": workspace_id, "exp": expire}
```
**规律**：多 workspace 系统中，workspace 上下文必须从 DB 权威来源获取，不能依赖 JWT 缓存字段（JWT 可能过期/不一致）。

## BE-21：SQLite 测试数据污染生产环境

**触发场景**：使用 curl/脚本对本地或生产环境测试业务逻辑（如频率限制、限额检查），测试结果写入了与生产共用的 SQLite 数据库文件。

**错误现象**：测试结束后，生产前端出现异常状态（如持续弹出确认弹窗、限额已用完），因为测试创建的脏数据被生产代码读取。

**根因**：测试用同一个 SQLite DB 文件（如 `~/.tashan-workbench/brain.db`），没有环境隔离。

**正确做法**：
```python
# 测试时使用临时 DB
import tempfile, os
os.environ["DB_PATH"] = tempfile.mktemp(suffix=".db")

# 或测试后清理（需在 conftest.py 或测试脚本末尾执行）
conn.execute("DELETE FROM brain_ask_requests WHERE question_draft LIKE '%test%'")
conn.execute("DELETE FROM users WHERE phone LIKE '139%'")
conn.commit()
```
**规律**：任何对有状态 DB 的测试，必须在测试开始前 setup 隔离环境，测试结束后 teardown 清理，或使用专用测试 DB 路径（`TEST_DB_PATH` 环境变量）。
