# Bug/产品偏差报告

> 填写时请严格区分：Bug（代码/逻辑错误）→ 发给开发；产品偏差（实现了但与PRD不符）→ 发给PM。

---

## Bug 报告

```markdown
### BUG-YYYYMMDD-NN: [简短描述]

**严重程度**：P0（阻塞核心路径）/ P1（功能错误）/ P2（体验问题）/ P3（细节）

**发现于功能模块**：[模块名]

**测试类型**：功能测试 / 闭环测试 / AI行为评测 / 体验验收 / 压力测试

---

**现象描述**：
[用户/测试者实际看到了什么，尽量附截图或错误日志]

**复现步骤**：
1. [具体操作步骤1]
2. [具体操作步骤2]
3. [结果：看到了什么]

**期望结果**：
[根据产品文档，应该发生什么]

**实际结果**：
[实际发生了什么]

---

**根因分类**（请勿混淆）：
- [ ] 代码/逻辑错误 → 发给：[具体开发者/模块]
- [ ] 产品偏差（与PRD不符）→ 发给：PM 决策

**相关错误日志**（如有）：
```
[粘贴相关日志]
```

**回归验证要求**：
修复后，需验证以下几点：
- [ ] [验证点1]
- [ ] [验证点2]
```

---

## 产品偏差报告

```markdown
### DEVIATION-YYYYMMDD-NN: [简短描述]

**发现于功能模块**：[模块名]

---

**产品文档（PRD）要求**：
> 引用产品定义.md 中的原文：
> "[具体要求]"

**当前实际实现**：
[描述代码/功能目前的实际行为]

**差距分析**：
- 设计要求：[X]
- 实际实现：[Y]  
- 差距：[Z]

---

**PM 决策选项**：

**选项A：修改实现，使其符合 PRD**
- 影响评估：开发工作量约 [X] 小时
- 需要修改：[具体文件/模块]

**选项B：调整 PRD，接受当前实现**
- 产品影响：[描述接受这个实现对用户体验的影响]
- 如接受，需在产品文档中更新：[具体章节]
```

---

**核心用户路径测试**：
✅ [步骤1] → [步骤2] → ... → [最终价值] 完整走通

---

**遗留 P2/P3 问题**（不阻塞上线，但记录）：

| # | 描述 | 严重程度 | 处理计划 |
