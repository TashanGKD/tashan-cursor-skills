## OPS-21：nginx location 前缀匹配会截走所有子路径，精确匹配才能选择性路由

**触发场景**：在 nginx 中把 `/api/auth/` 路由到外部服务，但某些 `/api/auth/` 子路径（如 `/api/auth/login`）需要保留在本服务处理。

**错误现象**：`location /api/auth/` 截走了所有 auth 子路径，连 login 和 /me 也路由到了外部服务，导致本产品特有的响应字段（如 workspace_id）丢失。

**根因**：nginx 的 `location /api/auth/` 是前缀匹配，匹配所有以 `/api/auth/` 开头的请求（包括 `/api/auth/login`、`/api/auth/me`）。

**正确做法**：
- 只路由特定端点用 `location = /api/auth/send-code`（精确匹配，`=` 前缀）
- 精确匹配的优先级高于前缀匹配，其余请求自然落到 `location /api/` 规则
- 示例（tashan-openbrain deploy.yml）：
  ```nginx
  location = /api/auth/send-code {
      proxy_pass http://127.0.0.1:8001/auth/send-code;
  }
  location = /api/auth/register {
      proxy_pass http://127.0.0.1:8001/auth/register;
  }
  location /api/ {
      proxy_pass http://127.0.0.1:8103/;  # 本产品后端
  }
  ```

**验证**：精确匹配后，send-code/register 到外部服务，login/me 到本产品 Python，workspace_id 正确返回。

## OPS-28：Python 3.9 不支持 `X | Y` union type 语法

**触发场景**：项目代码用了 `str | None`、`OpenAI | None` 等 union type 语法，部署到使用默认 Python 3.9 的系统时启动报 `TypeError` 或 `SyntaxError`。

**错误现象**：uvicorn 启动时抛 `TypeError: unsupported operand type(s) for |`，日志无法继续。

**根因**：`X | Y` union type 语法是 Python 3.10 新增的，Python 3.9 必须用 `Optional[X]` 或 `Union[X, Y]`。Cursor AI 默认生成 3.10+ 风格代码，部署到旧 Python 时直接炸。

**正确做法**：
1. 用 `uv` 创建指定版本 venv：`uv venv .venv --python 3.12`
2. 安装依赖：`uv pip install -r requirements.txt --python .venv/bin/python3`
3. 启动：`.venv/bin/python -m uvicorn app.main:app --port 8100`
4. 或在 pyproject.toml / .python-version 固定 `>=3.10`

**验证**：`uv venv --python 3.12` 后启动正常。

## OPS-29：.env 中 Windows 路径格式在 macOS/Linux 部署时失效

**触发场景**：.env 文件在 Windows 上生成（路径为 `T:\projects\...` 或 `C:\Users\...`），直接复用到 macOS/Linux 服务器时，Python settings.py 无法解析路径。

**错误现象**：`WORKSPACES_ROOT` / `SKILLS_DIR` 等路径变量解析报错，或指向根目录 `/T:/`（Linux 把 `T:\` 解析成 `T:/`）。

**正确做法**：
- 服务器部署前必须更新 .env 中所有路径变量为目标 OS 格式
- 推荐使用相对路径（`workspaces/`）或容器内的绝对路径（`/app/workspaces/`）
- 跨平台项目中路径变量一律使用正斜杠（`/`），Python `pathlib.Path` 在 Windows 上自动转换

## OPS-30：bore.pub SSH 隧道单端口仅允许 1 条并发 TCP 连接

**触发场景**：使用 bore.pub 暴露服务器 SSH 端口，AI 和用户同时尝试连接时其中一方被立即断开。

**错误现象**：AI 的 SSH 连接瞬间被关闭（Connection closed），无报错，像是被远端服务器拒绝，实际是 bore 并发限制。

**根因**：bore.pub 每个端口只允许 1 条并发 TCP 连接，用户占用时 AI 无法接入。

**正确做法**：
1. **AI 专用隧道**：在服务器上额外开一个 bore 隧道映射不同端口：
   ```bash
   bore local 2222 --to bore.pub  # AI 专用，不影响用户主连接
   bore local 22 --to bore.pub    # 用户主连接
   ```
2. **文件传输**：bore 不支持 SSH subsystem，`scp`/`rsync` 无法使用。改用 tar 管道：
   ```bash
   tar czf - files/ | ssh user@bore.pub -p PORT "cat > /tmp/transfer.tar.gz"
   ```
3. 等用户断开后 AI 再连接。

## OPS-31：PostgreSQL SERIAL sequence 在数据导入后未自动同步

**触发场景**：通过 pg_dump / INSERT 批量导入数据初始化 PostgreSQL 表（如 NPC 账号、初始用户），后续正常注册新用户时 500 Internal Server Error。

**错误现象**：INSERT 报 `duplicate key value violates unique constraint "users_pkey"`，说明 sequence 当前值低于表中已有的 MAX(id)。

**根因**：PostgreSQL SERIAL column 的 sequence 只在正常 INSERT 时自增；批量导入绕过了 sequence 自增机制，sequence 当前值停留在旧位置。

**正确做法**：
```sql
-- 导入数据后必须执行：
SELECT setval('public.users_id_seq', (SELECT MAX(id) FROM public.users));
-- 通用模式：对所有用批量导入的表执行
SELECT setval(pg_get_serial_sequence('表名', '列名'), MAX(列名)) FROM 表名;
```
**规律**：任何用非正常 INSERT 流程（pg_restore / COPY / 手动 INSERT VALUES）初始化的 PostgreSQL 表，启动后必须手动重置所有 SERIAL 列的 sequence。

## OPS-32：nginx `client_max_body_size` 默认 1MB，多文件上传失败

**触发场景**：前端一次上传多个文件（如多个 HTML/PDF 文档），nginx 反向代理转发给后端时返回 413 Request Entity Too Large 或直接断连，前端显示"上传失败"。

**错误现象**：单文件 < 1MB 时正常；多文件或单文件 > 1MB 时失败，后端收不到请求（nginx 在转发前就拒绝了）。

**根因**：nginx `client_max_body_size` 默认值为 1MB，新增的 server block 如果不显式设置，使用全局默认值。

**正确做法**：
```nginx
server {
    client_max_body_size 100M;    # server 级设置，覆盖全局默认
    
    location /api/ {
        proxy_pass http://127.0.0.1:8100/;
        client_max_body_size 100M;  # location 级双保险
    }
}
```
**排查**：`nginx -T | grep client_max_body_size` 确认各层配置；新建 server block 必须显式声明。

## OPS-33：Tauri 2.x 桌面应用打包专属流程（macOS）

**适用场景**：桌面应用（Tauri 2.x），不适用 Docker/nginx/服务器部署流程。

### macOS 26（Darwin 25.3）特有问题

| 问题 | 原因 | 解决方案 |
|---|---|---|
| `pnpm tauri build` 报 `CI=1` 相关错误 | 系统/CI 环境注入了 `CI=1`，Tauri 在 CI 模式下行为不同 | `env -u CI pnpm tauri build` |
| `bundle_dmg.sh` 参数缺失，DMG 打包失败 | Tauri 调用脚本时参数传递异常 | 跳过 DMG：`pnpm tauri build --bundles app`，再手动打 DMG |

### 手动打 DMG

```bash
# 1. 构建 .app
env -u CI pnpm tauri build --bundles app

# 2. 用 hdiutil 手动打 DMG
hdiutil create -volname "AppName" \
  -srcfolder "src-tauri/target/release/bundle/macos/AppName.app" \
  -ov -format UDZO \
  AppName-v1.0.dmg
```

### 桌面应用部署子流程（替代标准六步）

```
1. 品牌定制
   - Info.plist: bundle identifier / display name / version
   - src-tauri/tauri.conf.json: updater / icons / productName

2. 前端构建
   - pnpm build（确认输出到 src-tauri/frontend-dist/）

3. Tauri 构建
   - env -u CI pnpm tauri build --bundles app
   - 验证：dist/ 或 src-tauri/target/release/bundle/ 下有 .app

4. 验证 .app
   - 直接运行 .app，确认：① UI 正常 ② sidecar 后端启动 ③ 三端口就绪

5. 打 DMG（可选）
   - hdiutil create ...（见上方）

6. 更新 DEPLOY_ARCH.md
   - 记录版本号、打包日期、安装路径、sidecar 启动参数
```

### 注意事项

- Tauri sidecar 二进制的 token 与前端 `.env` 开发 token 不同步，初始化时前端可能 wsId=null（见 FE-17/BE-18）
- 非服务器应用不需要 CI/CD；用 GitHub Releases 或手动分发
- 生产环境测试：直接运行 `.app`，不用 `npm run dev`
