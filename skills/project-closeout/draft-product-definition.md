# project-closeout 产品定义草稿（Level 3，双视角）

---

## 一、人层设计

**解决的问题**：项目做完后再重新打开时，文档往往与实际决策发生漂移——有些用户决定没有记录到产品定义，有些技术架构与产品需求脱节，有些对话中的反馈从未被文档化。重新接手时 AI 无从判断「这个项目真实的完成状态」。

**触发条件（人的视角）**：
- 用户说：「做收尾检查」「确保文档自洽」「项目收尾」「项目关闭」「下次打开前检查一遍」
- 与 project-retrospective 不同：该 Skill 更新 Skill 体系；本 Skill 检查「项目文档与对话历史是否对齐」

**成功标准（人的视角）**：
- 输出一份「项目收尾报告」，明确：哪些文档不自洽、哪些用户输入没被文档化
- 有明确的后续动作清单（补录哪些文档、触发哪个 Skill）

**MVP 边界**：
- 检查 PM / Arch / Dev 三个角色的文档
- 扫描本项目的对话历史，识别未被文档化的用户输入
- 不做自动修复，只输出报告 + 路由建议

## 二、Agent 层设计

**触发条件（精确语义）**：
- 用户说包含「收尾」「关闭项目」「关闭前检查」「确保文档自洽」「下次打开」等词语
- 与 project-retrospective 区分：本 Skill 不更新 Skill 体系，只审查项目文档

**精确步骤序列**：

Step 1：用 explore 子智能体并行读取以下标准文档（若不存在则记录为「未找到」，不报错）：
- 产品经理/产品定义.md
- 技术架构师/技术架构.md
- 产品经理/开发计划.md
- 产品经理/产品问题追踪台.md
- 技术架构师/技术问题追踪台.md
- DEPLOY_ARCH.md
- 开发规范.md（可选）

Step 2：扫描本项目的对话历史
- 路径：agent-transcripts 目录下，按时间排序取最近 N 条（最多扫描 30 条，超过则标注「仅扫描最近30条，R10边界说明」）
- 分类用户输入为：需求/决策类 / 变更类 / Bug类 / 评价反馈类 / 其他

Step 3：并行调用三个审查子智能体
- /role-pm-auditor → 检查产品定义的完整性和一致性
- /role-arch-auditor → 检查技术架构与产品定义的对齐
- /role-dev-auditor → 检查开发计划与实际完成状态

Step 4：比对「用户输入分类清单」与「文档现状」
- 需求/决策类输入 → 产品定义.md 有没有对应记录？
- 变更类输入 → 相关文档有没有更新痕迹？
- Bug类输入 → 追踪台有没有对应条目？

Step 5：输出收尾报告到文件
- 路径：开发过程记录/项目收尾报告_YYYYMMDD.md

Step 6：路由后续动作

**边界条件**：
- 文档不存在 → 标注「未找到」，不阻断流程
- 对话历史超过 30 条 → 明确声明扫描边界（R10）
- 审查子智能体超时 → 标注「未完成」，继续其他步骤

## 三、人层 vs Agent 层张力分析

隐含假设（人觉得显然但 AI 需要被显式告知）：
- 「项目文档」有标准路径（产品经理/ 和 技术架构师/）
- 「对话历史」在 agent-transcripts 目录
- 「用户输入」指的是 role=user 的消息，不包括 AI 回复

可能产生歧义的点：
- 用户说「做收尾」，AI 可能同时触发 project-retrospective 和 project-closeout → 解决：触发词明确区分
- 「未被文档化的输入」是否包括很久以前的对话？→ 解决：只看最近 30 条，明确说明边界

## 四、失败模式

失败模式1：大量文档不存在（新建项目初期）
→ 根本原因：项目没有完整文档结构
→ 预防：Step 1 发现 2 个以上文档缺失时，提示「项目文档不完整，建议先完成文档后再运行收尾检查」

失败模式2：对话历史太长，扫描不完整
→ 根本原因：项目持续时间很长，对话超过 30 条
→ 预防：明确声明「已扫描最近 30 条对话，更早的历史未覆盖」（R10），用户可以手动扩展范围

失败模式3：三个审查子智能体发现问题太多，报告过大
→ 根本原因：项目积累了大量未解决的一致性问题
→ 预防：报告按「P0/P1/P2」分优先级，P0 必须立即处理，P1/P2 加入 issue-tracker

失败模式4：用户说「项目做完了」同时触发本 Skill 和 project-retrospective
→ 根本原因：触发词重叠
→ 预防：role-menu.mdc 中明确「项目收尾/收尾检查/确保文档自洽」触发本 Skill，「沉淀经验/更新 Skill」触发 project-retrospective

## 五、强绑定 Rule

强绑定（必须遵守）：
- R2 NO_FABRICATION：不得推断文档内容，必须实际读取
- R3 READ_FIRST：必须先读文档再做分析
- R10 MEMORY_BOUNDARY_EXPLICIT：对话历史扫描必须声明边界
- R6 ARTIFACT_FIRST：收尾报告必须写入文件，不能只在对话中输出

推荐注入：
- R7 CLOSED_LOOP_REQUIRED：报告中每个问题都要有后续动作建议

## 六、与现有组件的依赖关系

调用的组件：
- /role-pm-auditor（新建，只读）
- /role-arch-auditor（新建，只读）
- /role-dev-auditor（新建，只读）
- explore（内置，并行读取文档）
- issue-tracker（若发现 Bug 类漏记，路由到它）

影响的组件：
- project-retrospective（收尾报告完成后，建议紧接着运行 project-retrospective）
- role-menu.mdc（需要注册触发词，且明确与 project-retrospective 的触发词区分）
