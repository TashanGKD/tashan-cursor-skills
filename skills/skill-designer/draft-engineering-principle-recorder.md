## 一、人层设计（Human Layer）

### 1.1 解决什么问题？

```
当用户发现系统中存在原则性问题（如"LLM不应截断上下文"），
说一句"立一个原则"，AI 能自动分类该原则、写入正确的执行层、
找出当前代码/Skill中的违规点并生成修复计划——
而不是把它记到认知碎片本子里就没了。
```

### 1.2 触发条件（人的视角）

```
典型触发语句：
- 「立一个原则：X」
- 「记下这个原则：以后所有X都要Y」
- 「这是个原则，以后要遵守」
- 「把这记在相关文档里，以后要遵循」
- 「这应该是一个工程原则」

触发场景：
- 用户在执行某任务过程中发现一个反复违反的约束（如截断、硬编码）
- 用户想为 AI 系统建立一条新的行为规范（如"读文档不截断"）
- 用户想为工程设计立一条约束（如"LLM context 不截断"）

明确不应该触发的情况：
- 用户说"我注意到我有个思维习惯" → cognitive-extract-principle（思维/认知原则）
- 用户说"我倾向于从小点出发" → 认知原则，走 cognitive 路径
- 用户说"规范偏差：这里没按规范做" → project-convention-resolver
- 用户说"有个 bug" → issue-tracker
```

### 1.3 成功标准（人的视角）

```
产出物：
1. 原则被写入执行层（不是只记在日志或碎片里）
2. 当前代码/Skill/Rule 中的违规点被找出并记录为 issue
3. 用户收到一份「原则执行层覆盖报告」：在哪里记了、如何被强制执行

质量判断标准：
- 「两周后，另一个 AI session 执行相关任务时，会不会自动被这条原则约束？」
  → YES = 成功；NO = 失败
```

### 1.4 MVP 边界

```
✅ 做：
- 分类判断（4类：认知/AI行为/工程设计/代码违规）
- 对 A 类：路由到 cognitive-extract-principle
- 对 B 类：写入 session-bootstrap 或新 alwaysApply Rule（通过 skill-rule-修改规范）
- 对 C 类：写入最相关 Role Skill 的第一性原理节（通过 skill-rule-修改规范）
- 对 D 类：在代码库中搜索违规点，写入技术问题追踪台
- 输出覆盖报告

❌ 不做（MVP 范围外）：
- 自动执行代码修复（D类违规由用户决定是否触发 bug-fix-loop-coordinator）
- 追踪「原则被遵守的比率」（需要独立的监控机制）
- 一条原则同时录入三处以上（避免过度复杂化）
```

## 二、Agent 层设计（Agent Layer）

### 2.1 触发条件（AI 视角）

```
触发条件（精确描述）：
IF 用户消息语义包含：
  - "立一个原则" / "记下这个原则" / "这是个原则"
  - "以后[所有/每次/必须]X都要Y"（含"以后"+"规范性动词"）
  - "记在相关文档里"+"遵循/遵守/执行"
  - "应该记在原则里"
THEN 触发本 Skill

与其他组件的优先级：
- 若原则明确涉及"我的思维方式/判断习惯"（认知类） →
  → 在 Step 1 分类后路由到 cognitive-extract-principle，本 Skill 停止
- 若用户说"这里有个规范问题/这不符合规范"（规范偏差） →
  → 让 project-convention-resolver 优先处理（这是已知偏差，不是立新原则）
- 若用户说"有个 bug" →
  → issue-tracker 优先（bug != 原则）
```

### 2.2 执行步骤（精确序列）

```
Step 0  【读取必要上下文】
        Read: _内部总控/元项目导航.md（了解当前所有项目）
        Read: .cursor/skills/skill-index/SKILL-INDEX.md（了解可路由的 Skill）
        → 目的：为 Step 1 的路由决策准备基础知识

Step 1  【原则类型分类（必须与用户确认）】
        基于用户消息，初步判断原则类型：
        
        A. 认知/思维原则：
           判据：原则描述的是「郑总如何思考/判断/分析」
           示例：「先从小点出发升维」「推断前先验证」
           → 输出：「这条原则属于认知类，建议走 cognitive-extract-principle 流程」
           → 路由到 cognitive-extract-principle（本 Skill 完成）
        
        B. AI 行为约束：
           判据：原则描述的是「AI 执行任务时如何使用工具/如何操作」
           示例：「读文档不截断」「连接服务器前确认哪台机器」
           → 继续 Step 2-B
        
        C. 工程设计原则：
           判据：原则描述的是「代码/LLM调用/架构应该如何设计」
           示例：「LLM 调用不截断上下文」「认证必须复用现有服务」
           → 继续 Step 2-C
        
        D. 当前存在代码违规（C 的子集，或独立）：
           判据：用户指出「目前代码/Skill 里已经有违反这条原则的地方」
           → 在 Step 2-C 之后，继续 Step 2-D
        
        ⚠️ 多类型并存：一条原则可同时是 B+C+D，逐类处理，不跳过
        ⚠️ 不确定时：向用户确认「这条原则是关于AI系统行为，还是您自己的思维方式？」

Step 2-B  【AI 行为约束：写入执行层】
          判断应写入哪里：
          - 普遍约束（所有任务都适用）→ 建议写入 session-bootstrap.mdc 的合适节
          - 特定角色约束（只影响某些 Role）→ 写入对应 Role Skill 的约束节
          
          执行：
          → Read 目标文件（session-bootstrap 或 Role Skill）
          → 按 skill-rule-修改规范 三问（根因/影响范围/验证方法）
          → 执行备份 + StrReplace 写入 + 追加变更记录
          → 输出「B类录入完成，写入位置：[文件路径] [节标题]」

Step 2-C  【工程设计原则：写入最相关 Role Skill】
          判断最相关的 Role Skill（可多选）：
          - LLM/AI调用 → role-AI工程师/SKILL.md 第一性原理节
          - 部署/运维 → role-DevOps/REFERENCE.md 或 SKILL.md
          - 前端实现 → role-前端开发/SKILL.md
          - 后端实现 → role-后端开发/SKILL.md
          - 架构设计 → role-技术架构师/SKILL.md
          
          执行：
          → Read 目标 Skill 文件
          → 找到「第一性原理」节或等效约束节
          → 按 skill-rule-修改规范 备份 + 写入 + 变更记录
          → 输出「C类录入完成，写入位置：[文件路径] [节标题]」

Step 2-D  【代码违规审计 + 追踪】
          输入：用户描述的原则内容
          
          Step 2-D-1  构建搜索模式
                      基于原则内容，推导代码中可能的违规模式
                      示例：「不截断」→ 搜 `[:6000]` `[:3000]` `max_tokens=` `[:500]`
          
          Step 2-D-2  执行搜索
                      Grep: 项目代码目录（基于 project-config.md 确认路径）
                      → 找到所有匹配的违规位置
          
          Step 2-D-3  写入技术问题追踪台
                      → Read: 技术架构师/技术问题追踪台.md
                      → 对每个违规位置，追加一条 issue：
                        格式：| P1 | [文件路径:行号] | [违规描述] | 违反原则：[原则名] | 🔲 待修复 |
          
          Step 2-D-4  询问用户是否立即触发修复
                      → 「发现 N 处违规，已记录到追踪台。是否立即触发 bug-fix-loop-coordinator 修复？」
                      → 等待用户决策（不自动触发修复）

Step 3  【输出覆盖报告】
        格式：
        「✅ 原则录入完成
        原则：[原则内容]
        
        执行层覆盖：
        - [B类] [文件路径]：[如何被强制执行]
        - [C类] [文件路径]：[角色激活时必读]
        - [D类] 已记录 N 个违规到 [追踪台路径]
        
        可靠性评估：
        - 每次 [相关角色] 激活时必读约束 → 中高可靠
        - session-bootstrap alwaysApply → 高可靠
        
        如需最高可靠性：建议也写入 session-bootstrap B类节。」
```

### 2.3 边界条件

```
情况A：用户说的原则同时是认知类+工程类
  → 询问用户「这条原则同时适用于认知体系和工程体系，需要两路都处理吗？」
  → 根据用户决策分开处理

情况B：目标 Skill 文件没有「第一性原理」节
  → 在文件开头新建该节，再写入

情况C：代码违规搜索找不到匹配（原则是预防性的，当前无违规）
  → 只执行 Step 2-B/C 录入原则，跳过 Step 2-D
  → 在覆盖报告中注明「当前代码无违规，原则为预防性约束」

情况D：用户说的原则过于模糊
  → 先询问「这条原则的精确含义是什么？如何判断违反了它？」
  → 确认后再执行
```

### 2.4 依赖声明

```
依赖的其他 Skill/Agent/Rule：
- cognitive-extract-principle：Step 1 A类路由的目标
- skill-rule-修改规范：Step 2-B/C 的执行层，所有 Skill/Rule 文件的写入必须走它
- issue-tracker：Step 2-D 的参考格式（但直接写技术追踪台，不重复触发 issue-tracker）
- session-bootstrap.mdc：B类约束的主要写入目标之一

会影响的其他 Skill/Agent/Rule：
- 被写入的 Role Skill（role-AI工程师 等）：新增约束条目
- session-bootstrap.mdc：可能新增 B 类行为约束
- bug-fix-loop-coordinator：Step 2-D-4 可能被用户触发

认知根文档（L1/L1.5）：
- `_内部总控/认知结构/L1_系统性文档/系统架构思维维度/Skill体系设计原则_v1.0.md`
  关联：本 Skill 维护「Skill 体系执行约束」的完整性，与 Skill 体系设计原则中的「执行层」概念直接对应
```

### 2.5 常见失败模式

```
失败模式1：把工程原则误分到 cognitive-extract-principle
  → 根本原因：AI 看到「原则」就默认走认知路径
  → 预防方法：Step 1 必须先问用户/判断「这是思维方式还是系统行为」

失败模式2：只口头说「已录入」，没有实际工具调用写文件
  → 根本原因：AI 容易在 Step 3 只输出报告而没有真正执行 Step 2 的写入
  → 预防方法：Step 2 的每个执行动作必须有工具调用，Step 3 不先于工具调用输出

失败模式3：只写入一个地方，忽略其他需要同步的位置
  → 根本原因：AI 满足于「记下来了」，不追求「被可靠执行」
  → 预防方法：Step 3 覆盖报告必须主动评估是否需要写入更多执行层

失败模式4：代码违规搜索用错搜索词，错过真实违规
  → 根本原因：从原则语言到代码模式的映射不精确
  → 预防方法：Step 2-D-1 要列出多个候选搜索词，不止一个

回退动作：
- 若 Step 1 分类不确定 → 向用户确认，不猜测
- 若 skill-rule-修改规范 执行失败 → 暂停并告知用户，不继续后续步骤
```

### 2.6 强绑定 Rule

```
强绑定（必须遵守）：
- R2 NO_FABRICATION：不能说「已录入」而没有实际工具调用
- R6 ARTIFACT_FIRST：覆盖报告必须列出实际文件路径和节标题
- R3 READ_FIRST：写入前必须先读目标文件

推荐注入：
- project-doc-versioning-guard.mdc：写入 Skill 文件时需要备份
```

## 三、人层 vs Agent 层张力分析

```
人层设计中「不言而喻」的假设：
- 「写入执行层」= 实际改了文件，不是只说「我记下来了」
- AI 应该主动判断需要写哪些地方，不需要用户逐一指定

AI 可能产生歧义的点：
- 「第一性原理节」：这个节在不同 Skill 里名字和位置可能不同
  → 修订：Step 2-C 明确说「找到『第一性原理』节，如无则新建」
- 「覆盖报告」可能被 AI 理解为「说一段话」而非「工具调用之后的汇总」
  → 修订：Step 3 明确「此步骤在所有工具调用完成后执行」

两层对齐后的修订：
- Step 2-D-2 加入「多搜索词策略」
- Step 3 增加「工具调用顺序约束」：先 2-B/2-C/2-D 全部工具调用，最后才输出报告
```
