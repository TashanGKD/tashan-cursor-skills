## 一、人层设计

### 1.1 解决什么问题？

```
用户发现多个问题（Bug + 功能需求 + 原则 + 设计建议混合），
当前系统要么立即路由处理（打断节奏），要么遗漏部分意图。
本 Skill 提供「先全部存下来，需要时统一处理」的工作流：
任何混合输入 → 追加到项目待办文档 → 用户说「处理」时自动逐条修复。
```

### 1.2 触发条件

```
收集模式：
- 「有几个问题/反馈/建议，先记下来」
- 「我发现了以下问题：1. ... 2. ...」
- 「加到待办」「先记着」「以后处理」
- 用户发送含编号列表的混合反馈，且包含「先记」「存下来」等延迟信号

处理模式：
- 「处理待办」「开始修复待办」「逐条处理」
- 「按顺序修复」「自动处理所有待办」

明确不触发：
- 用户直接说「修复这个 bug」（只有单一意图，走产品开发域）
- 认知/思维类反思 → cognitive-extract-principle
```

### 1.3 成功标准

```
收集模式：
  产出：项目待办.md 中新增若干条 🔲 待处理条目
  质量：每条条目有类型标签、内容摘要、来源日期

处理模式：
  产出：所有 🔲 条目变为 ✅ 或 ❌
  质量：用户一次指令后全程无需干预（除必要决策点外）
```

### 1.4 MVP 边界

```
✅ 做：
- 解析混合输入为结构化条目（最多 20 条）
- 按类型标记（Bug/Feature/UX/原则/设计建议）
- 追加到项目待办.md（格式化表格）
- 处理时：原则先于 Bug，P0 先于 P1，设计建议仅存档
- 前置决策收集：B 类（功能/架构决策）一次性问完再执行
- 执行后更新条目状态

❌ 不做（MVP 不含）：
- 多项目待办合并管理
- 跨 session 的优先级重排
- 待办条目之间的依赖关系建模
```

## 二、Agent 层设计

### 2.1 触发条件（AI 视角）

```
收集模式触发：
IF 消息语义包含 「先记/存下来/待办/记录一下/以后处理」
   AND 消息包含 ≥2 个独立意图（编号列表 OR 多个独立句子）
THEN 触发收集模式

处理模式触发：
IF 消息语义包含「处理待办/修复待办/逐条处理/开始处理/自动处理所有」
THEN 触发处理模式

优先级：
- 若「立一个原则」+ 其他意图混合 → 先收集到待办，原则条目在处理阶段路由到 engineering-principle-recorder
- 若只有单一意图且不含延迟信号 → 走 A1 正常路由，不触发本 Skill
```

### 2.2 执行步骤

```
══════════════ 收集模式 ══════════════

Step C-0  【读取待办文档路径】
          Read: .cursor/project-config.md（若存在）→ 获取 PATH_待办
          默认路径：项目群/[当前项目]/产品经理/项目待办.md
          若文件不存在 → 创建并写入标准表头

Step C-1  【解析输入为结构化条目】
          读取用户消息，识别所有独立意图
          对每条意图：
            - 提炼一句话摘要（≤40字）
            - 标记类型：Bug / Feature / UX / 原则 / 设计建议
            - 估计优先级：
              * Bug = P0（阻塞）/ P1（影响核心）/ P2（次要）
              * 原则 = 不设优先级
              * Feature/UX = 按用户描述语气估计
              * 设计建议 = 低优先级（存档）

Step C-2  【追加到项目待办.md】
          Read: [PATH_待办]（先读确认格式和现有条目 ID）
          StrReplace 追加新行：
          | B-NN | [类型] | [摘要] | [YYYY-MM-DD] | [P0/P1/P2/—] | 🔲 |
          ID 格式：B=Bug / F=Feature / U=UX / P=原则 / D=设计建议 + 两位序号

Step C-3  【确认输出】
          「✅ 已记录 N 条待办到项目待办.md：
          [B-01] [Bug/P1] 更新认知显示正在构建中
          [P-01] [原则] LLM 调用不截断上下文
          ...
          说『处理待办』可触发自动处理流程。」

══════════════ 处理模式 ══════════════

Step P-0  【读取待办文档 + 当前项目上下文】
          Read: [PATH_待办]
          提取所有 🔲 条目，按以下顺序排序：
            1. 原则类（影响所有后续修复方向）
            2. P0 Bug
            3. P1 Bug
            4. UX 问题（已有决策方向）
            5. Feature（已有决策方向）
            6. P2 Bug / 设计建议（最后）

Step P-1  【前置决策收集（B 类条目）】
          扫描所有 🔲 条目，找出需要用户决策的项：
            - Feature 类：有多种实现方案时（询问方案选择）
            - UX 类：设计方向不明时（询问期望体验）
            - 架构类设计建议：询问是否纳入当前迭代
          
          ⚠️ 一次性集中提问（不分散，不在执行中途打断）：
          「处理前需要您确认以下决策点：
           [F-01] 实时显示AI思考过程 → A: SSE推送 / B: 轮询？
           [D-01] 状态管理重构 → 纳入当前迭代 / 下个迭代？」
          
          等待用户全部回答后，进入 P-2 自动执行

Step P-2  【逐条自动执行】
          ⚠️ 执行期间不再打断用户，除非遇到完全无法判断的情况

          对每条 🔲 条目，按顺序执行：

          [原则类] → 触发 engineering-principle-recorder
                    等待完成 → 更新状态 ✅

          [Bug/P0] → 直接触发 bug-fix-loop-coordinator
                    设置参数：仅处理此条 Bug
                    等待完成 → 更新状态 ✅

          [Bug/P1/P2] → 写入技术问题追踪台
                        触发 bug-fix-loop-coordinator 处理
                        等待完成 → 更新状态 ✅

          [Feature/UX（已有决策）] → 触发 role-产品开发协调者
                                    传入已决策内容
                                    等待完成 → 更新状态 ✅

          [设计建议] → 追加到技术架构.md 或捕捉为认知碎片（不立即执行）
                      更新状态 ✅（存档完成）

Step P-3  【输出批次处理报告】
          「✅ 本批次处理完成
          ─────────────────────────
          [P-01] 原则：LLM不截断 → 已写入 role-AI工程师 第一性原理
          [B-01] Bug/P1：更新认知显示构建中 → 已修复，提交 xxx
          [F-01] Feature：实时可视化 → 已进入产品设计流程
          [D-01] 设计建议：状态管理 → 已存档到技术架构.md
          ─────────────────────────
          待处理：0 条 | 完成：N 条
          完整记录见：[PATH_待办]」
```

### 2.3 边界条件

```
情况A：PATH_待办 文件不存在
  → Step C-0 自动创建，写入标准表头后继续

情况B：Step P-1 用户没有回答决策问题（说「跳过」）
  → 未决策的 B 类条目标记为 ⏸️ 搁置，跳过继续处理其他条目

情况C：某条目执行失败（下游 Skill 执行报错）
  → 条目标记为 ❌ 执行失败（原因），继续下一条
  → 在 Step P-3 报告中列出失败项

情况D：处理模式下待办文档为空或无 🔲 条目
  → 告知用户「当前没有待处理条目」，不执行任何操作
```

### 2.4 依赖声明

```
依赖的其他 Skill/Agent/Rule：
- engineering-principle-recorder：处理原则类条目
- bug-fix-loop-coordinator：处理 Bug 类条目
- role-产品开发协调者：处理 Feature/UX 类条目
- role-技术架构师 / cognitive-capture-fragment：处理设计建议类条目

会影响的其他 Skill/Agent/Rule：
- session-bootstrap A1：需要新增排除条件（多意图收集信号不走产品开发域）
- role-menu.mdc：需要注册本 Skill
- 各项目的技术问题追踪台.md：Bug 类条目处理时写入

认知根文档：
- `_内部总控/认知结构/L1_系统性文档/系统架构思维维度/Skill体系设计原则_v1.0.md`
  关联：本 Skill 实现「文档即状态」的执行协议，与设计原则中「执行层文档管理」概念对应
```

### 2.5 常见失败模式

```
失败1：收集模式和处理模式混触
  → 根因：「处理这几个问题」既像「处理待办」又像「收集多个问题」
  → 预防：Step C-0 触发前检查消息是否包含「处理待办/修复」等处理信号
         含处理信号 → 处理模式；含延迟信号 → 收集模式；两者都无 → 向用户确认

失败2：执行期间中途问用户
  → 根因：下游 Skill 的某个步骤需要用户决策
  → 预防：Step P-1 前置决策收集覆盖所有已知决策点；执行中遇到新决策点 → 记录为 ⏸️ 不打断

失败3：收集时遗漏某些意图
  → 根因：AI 解析混合输入时漏掉了隐含的意图
  → 预防：Step C-1 要求"对每条句子逐一判断是否为独立意图"，有疑问时宁多勿少
  
失败4：项目路径错误（待办文档存错位置）
  → 根因：project-config.md 路径未配置，默认路径猜错项目
  → 预防：Step C-0 明确读 project-config.md 或询问用户「当前是哪个项目？」
```

### 2.6 强绑定 Rule

```
强绑定：
- R2 NO_FABRICATION：不能说「已处理」而没有实际工具调用
- R6 ARTIFACT_FIRST：项目待办.md 是本 Skill 的核心产出物，必须实际写入

推荐注入：
- project-doc-versioning-guard.mdc：修改项目文档时执行备份
```

## 三、人层 vs Agent 层张力

```
隐含假设：用户说「有几个问题」一定意味着「先收集不立即处理」
  → AI 可能将其理解为「立即修复」→ 触发产品开发域
  → 修订：必须有「延迟信号词」才触发收集模式（见 2.1）

隐含假设：设计建议等等于「存档」
  → 部分设计建议用户可能想立即讨论
  → 修订：设计建议类在 P-1 前置决策中增加询问「是否需要立即讨论？」

两层对齐修订：
- 触发词需要显式延迟信号（「先记」「待办」「以后处理」）
- 纯「有问题」不触发，避免与产品开发域冲突
```
