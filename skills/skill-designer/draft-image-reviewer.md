## 一、人层设计（Human Layer）

### 1.1 解决什么问题？

```
批量生成的候选图需要被「看见」后才能选择，
AI 先扮演挑剔读者自审一轮（关卡C结构），
再生成一个用户可以直接打开的 MD 审核文件——图并排展示、意见有格子写——
用户选图、提意见、AI 按意见修改迭代，直到定稿并写回手稿。
整个过程有历史留痕，每轮迭代保存不覆盖。
```

### 1.2 触发条件（人的视角）

```
典型触发语句：
- 「审核候选图」
- 「生成审核文件」
- 「帮我看看哪张图好」
- 「图生完了，做一下审核」
- 「图有问题，重新审一轮」

触发场景：
- image-generator 完成后，用户准备选图
- 用户已有一组图片（不一定通过 image-generator 生成），需要审核和选择
- 某轮审核后用户提了意见，重新生成图后再次审核（iteration）

明确不应该触发的情况：
- 用户只想看图（直接打开目录查看即可）→ 不需要 Skill，直接查看文件
- 用户想要 AI 直接替他决定用哪张（「你帮我选一张吧」）
  → 本 Skill 提供推荐，但最终选择权在用户；AI 不单方面决定
```

### 1.3 成功标准（人的视角）

```
产出物：
- review-[位置标识]-v[N].md（审核文件，所有候选图内嵌，可直接在 MD 预览器打开）
- 用户完成选择后：
  - 手稿中对应段落更新为选定图的引用和提示词记录
  - 落选图归档到 archive/ 目录（不删除）
  - 审核文件本身归档留痕

质量判断标准：
- 打开审核 MD，能直接看到所有候选图
- 每张图旁边有视角说明和 AI 自审意见（不是空格）
- 审核文件有明确的「选择格」，用户知道在哪里写意见
- 用户提意见后，下一个版本能追踪到哪张图对应了哪条意见的修改
```

### 1.4 MVP 边界

```
✅ 做：
- AI 自审（挑剔读者视角）：对每张候选图给出评估
- 生成可直接打开的 review MD（图内嵌 + 意见区）
- 解析用户在审核文件里或对话里写的意见
- 路由意见：
  - 若意见指向提示词修改 → 回 image-ideation（修改视角）或直接修改提示词
  - 若意见指向风格调整 → 修改风格后重调 image-generator
  - 若确认选定 → 执行定稿入稿
- 定稿入稿：更新手稿 MD、归档落选图和审核文件

❌ 不做：
- 不自动决定最终选哪张（选择权归用户）
- 不修改图片本身（裁剪/调色等后处理）→ 另立工具
- 不评估文字内容对不对（那是 article-proofreading 的事）
- 不处理手稿中除图片引用以外的内容修改
```

## 二、Agent 层设计（Agent Layer）

### 2.1 触发条件（AI 视角）

```
触发条件（精确描述）：
IF 用户说「审核候选图」/ 「生成审核文件」/ 「帮我看看图」
   AND（存在 generation-log 文件 OR 用户指定了图片目录）
THEN 触发本 Skill

与其他组件的优先级：
- 本 Skill 在 image-generator 之后；若 generation-log 找不到，
  询问用户图片位置后继续（兼容手动生成的图片场景）
- 若用户说「这张图有问题，重新画」→ 先触发 image-ideation/image-generator，
  完成后自动触发本 Skill 进行新一轮审核
```

### 2.2 执行步骤（精确序列）

```
Step 1  读取候选图清单
        输入：generation-log 文件路径（优先）
              OR 用户指定的图片目录
        操作：获取所有候选图列表（含：文件路径、视角名、提示词记录）
        输出：[内部] 候选图元数据列表
        失败时：若找不到图片文件，告知用户确认路径

Step 2  AI 挑剔读者自审（关卡C结构，不可跳过）
        输入：每张候选图 + 对应的视角叙事假设（从 ideation 文件读取）
        操作：切换到「挑剔读者」视角，对每张图独立评估：
              评估维度：
                a. 关系揭示：图是否真正展示了叙事假设里的「关系/结构」？
                b. 信息增量：去掉这张图，文章意思还完整吗？图有没有额外价值？
                c. 可读性：5秒内能抓住核心信息吗？
                d. 逻辑一致性：图中若有文字/标注，与文章论点一致吗？
              评估结论：「强烈推荐」/ 「可用但有问题：[具体问题]」/ 「不推荐：[原因]」
        输出：每张图的 AI 自审评语（3-4 句，包含结论等级 + 具体说明）
        失败时：若无法查看图片（格式不支持），标注「[AI无法评估，请用户自行判断]」

Step 3  生成审核 MD 文件
        输入：候选图列表 + AI 自审评语
        操作：按格式生成 review MD 文件（见格式规范）
              文件路径：[文章目录]/images/review/
                        review-[位置标识]-v[版本号].md
              核心要求：
              - 每张图用 HTML img 标签内嵌（确保 MD 预览器可直接显示）
              - 每张图旁有：视角名 + AI自审评语 + 用户意见区（空白）+ 选择框
              - 文件顶部有操作指南（告诉用户怎么在文件里写意见）
        输出：review MD 文件
        失败时：若图片路径包含空格或特殊字符导致无法内嵌，转换为相对路径

Step 4  告知用户并等待反馈
        输出：「✅ 审核文件已生成：[路径]
               AI 推荐：[视角X]（[一句话理由]）
               请打开文件，在每张图下方的「用户意见」区填写反馈，
               或直接在这里告诉我：
               「选[视角X]」 → 定稿入稿
               「[视角X]有问题：[描述]」 → 修改后重生成
               「所有图都不满意」 → 回到 image-ideation 重新构思」
        等待用户输入

Step 5  解析用户反馈并路由
        输入：用户在对话中的反馈（或 AI 读取用户修改后的 review MD）
        操作：判断反馈类型：
              A. 「选 [视角X]」 → 进入 Step 6（定稿入稿）
              B. 「[视角X] 的 [问题描述]」 → 
                 - 若问题是提示词/视角层面 → 回 image-ideation，传入意见
                 - 若问题是风格/颜色层面 → 直接修改提示词，回 image-generator
                 - 意见写入 iteration-log（见附录）
              C. 「所有图都不满意」→ 回 image-ideation 从头重构思
        输出：路由决策 + 下一步行动说明
        失败时：若无法判断用户意图，追问一个问题：
               「你希望修改[哪个方面]，还是选定某张图直接用？」

Step 6  定稿入稿（用户确认选定后执行）
        输入：选定的图片文件路径 + 手稿 MD 文件路径 + 位置标识
        操作：
          a. 在手稿 MD 对应段落插入/替换图片引用（HTML img 标签或 Markdown 格式）
          b. 在手稿对应段落下方更新提示词记录区（按发布手册 §7.1 格式）
          c. 落选图移动到 [文章目录]/images/archive/ 目录
          d. 审核文件（review MD）移动到 [文章目录]/images/review/archive/
          e. 更新 generation-log：标注「最终选定：[文件名]，日期：YYYY-MM-DD」
        输出：「✅ 已定稿：
               手稿更新：[手稿路径] 第[N]段
               已归档：[N]张落选图 + 本轮审核文件
               历史留痕：[archive 路径]」
        失败时：若手稿文件定位失败（找不到对应段落），
               告知用户手动确认插入位置后继续
```

```markdown

> 版本：v[N] | 审核日期：YYYY-MM-DD | 候选图数量：N
> 上游：generation-log-[位置]-v[N].md
> 操作指引：在每张图的「用户意见」行填写选择或反馈

  <img src="../candidates/[文件名].png" style="max-width:80%">
</p>

**视角叙事假设**：[来自 ideation 文件]

**AI 自审评语**：[等级] - [具体评估，3-4句]

**用户意见**：[请在这里填写：选用 / 不选，原因：___]

---

## 候选图 2：[视角名]
...

---

## 迭代记录

| 版本 | 日期 | 变更说明 | 修改的图 |
|------|------|---------|---------|
| v1 | YYYY-MM-DD | 初始生成 | - |
| v2 | YYYY-MM-DD | [视角X]提示词修改：[原因] | [文件名变化] |
```

### 迭代记录 iteration-log（附录）

每次意见导致重新生成，追加以下记录：

```markdown
## 迭代 [N]：YYYY-MM-DD

**意见来源**：用户 / AI自审
**针对图片**：[文件名]
**意见内容**：[原文]
**路由到**：image-ideation（修改视角）/ image-generator（修改风格）
**修改摘要**：[提示词哪里变了]
**新图文件**：[新文件名]
```

### 2.3 边界条件

```
情况A：用户打开 review MD 后直接在文件里填写了意见，没有回到对话
       → AI 在下次触发时先读取 review MD，检查是否有填写的「用户意见」行
         若有，作为本轮输入解析

情况B：多次迭代后图片数量超过 10 张（多轮 A/B 变体累积）
       → 在审核文件顶部加警告：「本位置已迭代 N 轮，建议选定或重新从 ideation 构思」
         不强制停止，但提醒

情况C：用户想同时保留两张图（不只选一张）
       → 支持：询问「两张图分别放在哪个位置？」然后各自入稿

情况D：同位置已有 review-v1/v2，再次触发时
       → 自动创建 review-v3，不覆盖旧版（历史留痕）

情况E：用户说「AI 帮我选」
       → AI 给出推荐理由，但明确：「最终由你确认，说「确认选[X]」后执行入稿」
```

### 2.4 依赖声明

```
依赖的其他 Skill/Agent/Rule：
- image-generator：消费其 generation-log 和候选图文件
- image-ideation：读取其 ideation 文件以获取叙事假设（用于 Step 2 自审）
- wechat-article-writer / 通用写作者角色：Step 6 回写的目标手稿
- 发布手册 §7.1：Step 6 入稿时的提示词记录格式

会影响的其他 Skill/Agent/Rule：
- 文章手稿 MD：Step 6 直接修改
- image-ideation / image-generator：意见路由时触发它们重新执行
```

### 2.5 常见失败模式

```
失败模式1：AI 自审只写了「很好」「推荐」，没有具体说明
  → 根本原因：Step 2 没有强制四维评估结构，AI 走了捷径
  → 预防方法：Step 2 明确四个评估维度（关系揭示/信息增量/可读性/逻辑一致性）
              每维度必须有一句话，不允许只写「好」

失败模式2：review MD 图片显示不出来（路径错误）
  → 根本原因：使用了绝对路径，在不同环境下失效
  → 预防方法：Step 3 强制使用相对路径（../candidates/xxx.png）

失败模式3：定稿入稿后，手稿里的图出现了两张（旧图没有被替换）
  → 根本原因：Step 6 做了「插入」而不是「替换」
  → 预防方法：Step 6 明确：先检查对应位置是否已有图片引用，有则替换，无则插入

失败模式4：迭代多轮后无法追溯哪条意见导致了哪次修改
  → 根本原因：iteration-log 没有被维护
  → 预防方法：每次意见路由（Step 5B）立即写入 iteration-log，不在 Step 6 之后补

回退动作：
- 若入稿失败（文件定位错误）→ 输出 HTML img 标签，用户手动插入
- 若用户迭代超过 3 轮仍不满意 → 建议「重新从 image-ideation 构思，彻底换视角」
```

### 2.6 强绑定 Rule

```
强绑定（必须遵守）：
- R2 NO_FABRICATION：AI 自审不能捏造对图的评估（「图很好」不是评估）；
                     只能基于实际看到的图内容评价
- R8 VERIFY_BEFORE_FREEZE：入稿前必须确认手稿路径和段落正确，不可假设
- R9 VERSION_CONTINUITY：review MD 版本号连续，每轮迭代产生新版本，不覆盖旧版

推荐注入：
- R6 ARTIFACT_FIRST：每次审核生成 review MD 文件，不接受只在对话里口头说「这张好」
- R13 RULE_BEATS_STYLE：AI 自审的诚实评估（「这张图信息增量为零」）
                         比「给用户正面反馈」更重要
```

## 三、人层 vs Agent 层张力分析

```
人层设计中「不言而喻」的假设：
- 人觉得「直接打开 MD 就能看到图」很自然
  → AI 需要知道：Cursor IDE 的 MD 预览支持相对路径 img 标签，
    但如果路径包含中文或空格，需要 URL 编码；必须测试一种可用的嵌入方式
- 人觉得「审核者视角」很清晰
  → AI 需要被明确告知：不是泛泛地「这张图好」，
    而是从四个具体维度出发，每维度给一句实质性判断

AI 可能产生歧义的点：
- 「挑剔读者」可能被 AI 理解为「比普通读者更苛刻」，仍然写正面评价
  → 已在 Step 2 中明确：评估核心问题是「信息增量是否存在」，这是二元判断，不模糊
- 「定稿入稿」的「对应位置」可能 AI 找不到
  → 已加失败处理：找不到时让用户手动确认

两层对齐后的修订：
- Step 2 加了「四维评估」强制结构，防止 AI 走捷径
- Step 6 明确「替换」而非「插入」，防止重复图片
- Step 5 加了「无法判断意图时追问一个问题」约束，防止猜测用户意图
```
