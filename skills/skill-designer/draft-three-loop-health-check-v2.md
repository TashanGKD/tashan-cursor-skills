## 一、人层设计（Human Layer）

### 1.1 解决什么问题？

```
现有体系有很多 Skills/Roles/Rules，但没有一个机制能回答：
「这套体系，在真实使用场景下，从头到尾能不能跑通？」

具体来说，它能回答三个问题：
① 这个系统实际支撑了哪些任务场景？（场景发现）
② 对每个场景，完整的闭环链路健康吗？有没有断裂节点？
③ 每个场景里，认知体系有没有被嵌入？（认知根 + 认知反馈）
```

### 1.2 触发条件（人的视角）

```
典型触发语句：
- 「三大闭环健康检查」「全系统场景验证」
- 「检查一下现在体系健不健康」「系统自检」
- 「场景化闭环检查」

触发场景：
- 长时间体系演进后，想确认整体是否仍然自洽
- 新增了多个 Skill 后，想验证各场景的完整性
- 发现某个场景似乎有问题，想做完整验证

明确不应该触发的情况：
- 「Skill 体系内部一致性」→ skill-system-health-check
- 「认知结构 C1-C11」→ cognitive-consistency-check
- 「某个域的传播完备性」→ skill-domain-health-check
```

### 1.3 成功标准（人的视角）

```
产出物：场景级健康报告（对话输出 + 可选生成沙盘文件）

质量判断标准：
- 发现的所有场景都有明确的「闭环是否健康」判断
- 每个断裂节点有具体的「在哪里断」和「怎么修」说明
- 每个场景有「认知根」和「认知反馈路径」的明确检查结论
```

### 1.4 MVP 边界

```
✅ 做：
- 动态场景发现（从现有体系归纳，不预设）
- 每个场景的最小闭环 + 大闭环识别
- 全量场景的沙盘推演（Phase 1 预想 + Phase 2 实际验证）
- 每个沙盘内置认知根检查 + 认知反馈检查
- 生成场景级健康报告

❌ 不做：
- 自动修复（只诊断，不修复）
- 实时执行真实任务来验证（沙盘是静态分析，不是实际运行）
- 替代 skill-closure-verifier-meta 的节点级传播完备性检查
```

## 二、Agent 层设计（Agent Layer）

### 2.1 触发条件（AI 视角）

```
触发条件（精确）：
用户消息包含：（「三大闭环」OR「全系统」OR「场景化」）AND（「健康」OR「检查」OR「验证」OR「自检」）

优先级：
- 「Skill 体系内部一致性/触发词冲突/版本漂移」→ skill-system-health-check 优先
- 「C1-C11 认知自洽」→ cognitive-consistency-check 优先
- 「元验证/沙盘验证传播」→ skill-closure-verifier-meta 优先
- 其余（全场景健康）→ 本 Skill
```

### 2.2 执行步骤（精确序列）

```
Step 1  动态场景发现
        输入：role-menu.mdc + SKILL-INDEX.md + DOMAIN-REGISTRY.md
        
        1a. Read: role-menu.mdc → 提取所有 role-* 条目（目录名 + 触发场景描述）
        1b. Read: SKILL-INDEX.md → 读取 Skills 的功能说明（description/备注列）
        1c. Read: _内部总控/skill-system-design/DOMAIN-REGISTRY.md → 读取已定义工作域
        
        1d. 归纳任务场景算法：
            - 把 role-* Skills 按「它做的是哪类工作」聚类
            - 参考 DOMAIN-REGISTRY 已有定义（不强制对齐，只参考）
            - 输出：发现的场景列表（每个场景：名称 + 涉及的 Skills/Roles 列表）
            - 示例可能发现：产品开发 / 内容创作 / Skill体系管理 / 认知结构管理 / 科研画像 / 公司运营
            - 实际数量由扫描决定，不预设
        
        输出：discovered_scenarios（动态列表，附涉及组件）
        失败处理：任一文件不可读 → 标注「场景发现不完整」，继续其他步骤

Step 2  对每个场景识别闭环结构
        对 discovered_scenarios 中每个场景：
        
        2a. 读取该场景涉及的核心 Skill 的 SKILL.md（激活后立即执行步骤）
        2b. 识别「最小闭环」：
            - 触发条件 → 执行步骤 → 终态（工作产物存在）
            - 最短的完整一圈
        2c. 识别「大闭环」（在最小闭环基础上扩展）：
            - D0 节点：任务开始时，是否有步骤读取相关 L1 认知文档？（认知根）
            - 末端节点：任务完成后，是否有步骤将新洞见写入 L2？（认知反馈）
            - 级联节点：任务产物是否触发了下一个闭环的输入？
        
        输出：per-scenario loop structures（每个场景的闭环链路图）

Step 3  为每个大闭环设计沙盘
        对每个场景的大闭环，生成沙盘（内存中，可选保存文件）：
        
        沙盘结构（遵循 SANDBOX-FORMAT 精神，场景级适配）：
        
        [场景名称] 沙盘
        ├── 输入：[真实触发情境，一句话]
        ├── Phase 1 预想链路：
        │   [触发] → [Step1输出] → [Step2输出] → ... → [终态]
        │   含：D0认知根检查节点、L2写入节点、级联触发节点
        ├── 分支覆盖：[正常路径 / 失败路径 / 认知路径]
        └── 预期终态：[具体的工作产物 + 认知产物]
        
        输出：scenario_sandboxes（每个场景一个沙盘，含预期结果）

Step 4  执行 Phase 2 验证（对照现有体系）
        对每个 scenario_sandbox，逐节点验证：
        
        4a. 节点存在性检查：
            - 每个预期的 Skill/Role/Rule 是否在 SKILL-INDEX 中存在？
            - 版本是否稳定（不是「待验证」状态的核心节点）？
        
        4b. 输入输出连接检查：
            - Step N 的输出（文件/状态），是 Step N+1 的有效输入吗？
            - 特别检查：关卡A/B/C 的前置条件是否被满足？
        
        4c. 认知根检查（D0 节点验证）：
            - 该场景的核心 role-* Skill 的知识导航表里，是否有 D0 行？
            - D0 行指向的 L1 文档是否存在？
        
        4d. 认知反馈检查（末端节点验证）：
            - 任务完成后（session-bootstrap 序列B），是否有机制将洞见写回 L2？
            - write-task-log 步骤五（cognitive-task-reflector）是否会被触发？
        
        4e. 文档自洽检查：
            - 该场景涉及的核心文档（产品定义/技术架构/任务日志等）
              是否有一致的版本标注和变更记录机制？
        
        输出：per-scenario validation results（每个节点：✅/❌/⚠️ + 证据）

Step 5  生成场景级健康报告
        
        「━━━━━━━━━━━━━━━━━━━━━━━━━━
          📋 全场景健康检查报告
          日期：YYYY-MM-DD
          发现场景数：N
          ━━━━━━━━━━━━━━━━━━━━━━━━━━
          
          ── 按场景汇总 ──
          [场景1]：✅/⚠️/❌ [一句话摘要]
            · 最小闭环：[通过/N个断裂]
            · 认知根：✅/❌/⚠️
            · 认知反馈路径：✅/❌/⚠️
          [场景2]：...
          
          ── 按缺口类型汇总 ──
          缺失节点：N 个（Skill/Role 不存在）
          断裂连接：N 个（输入输出不匹配）
          无认知根（D0缺失）：N 个场景
          无反馈路径：N 个场景
          文档自洽问题：N 个
          
          ── 优先修复清单（按影响面排序）──
          P0: [描述] → 修复建议：[Skill/步骤]
          P1: [描述] → 修复建议：...
          ...
          ━━━━━━━━━━━━━━━━━━━━━━━━━━
          
          [生成沙盘文件] [处理第1项 P0 问题]」
```

### 2.3 边界条件

```
情况A：发现的场景数 > 10，验证耗时过长
  → 告知用户「发现 N 个场景，完整验证预计需要大量操作。是否优先验证核心场景？」
  → 等用户确认

情况B：某个场景的 SKILL.md 文件不可读
  → 标注「[场景X] 部分节点无法验证（文件不可读）」，继续其他场景

情况C：某个场景的大闭环无法识别（步骤过于模糊）
  → 标注「闭环结构不清晰，需要 Skill 步骤更精确化」，列为 ⚠️

情况D：认知根文档（D0指向的L1文档）不存在
  → 标注「认知根文档缺失（路径：[路径]）」，列为 ❌
```

### 2.4 依赖声明

```
读取（不修改）：
- role-menu.mdc（场景发现）
- SKILL-INDEX.md（节点存在性检查）
- DOMAIN-REGISTRY.md（参考已有域定义）
- 各 role-*/cognitive-*/内容创作 Skill 的 SKILL.md
- L0_大脑总地图.md（认知根文档存在性检查）

不调用其他 Skill（独立执行，避免 session-bootstrap 级联）
不写入任何文件（纯诊断，可选输出到沙盘文件）

认知根文档（L1/L1.5）：
- 三大闭环架构蓝图.md（「任意外部输入的自运转能力」§三）
- AI智能体任务分类体系与闭环完成标准_v1.0.md（任务分类框架）
- Skill体系设计原则_v1.0.md §4.3.5（认知根原则）
```

### 2.5 常见失败模式

```
失败模式1：Step 1 归纳场景时，把能力层/子任务 Skill 也算进场景
  → 根本原因：SKILL-INDEX 中 Skill 分类边界不够清晰
  → 预防：Step 1 只看 role-* Skills 聚类，不看能力层/子任务 Skill

失败模式2：Step 2 识别闭环时，只识别了执行步骤，遗漏了认知节点
  → 根本原因：认知根/反馈是「应该有」而非「明文有」，AI 可能忽略
  → 预防：Step 2 有强制 4c/4d 检查项，不可跳过

失败模式3：Step 4 验证时，AI 自我声称「节点存在」但未实际读文件
  → 根本原因：R2 NO_FABRICATION 违反
  → 预防：所有「节点存在性」判断必须基于 SKILL-INDEX 实际内容，不得凭记忆

失败模式4：发现场景太多，逐一验证导致 context 爆炸
  → 根本原因：全量验证对大型体系确实是重操作
  → 预防：Step 4 可分批执行（每次处理 2-3 个场景），完成后汇总

回退动作：
- 任何 Step 失败 → 标注「此场景验证不完整」，继续其他场景
- 不允许「验证了部分」声称「全量通过」
```

### 2.6 强绑定 Rule

```
强绑定（必须遵守）：
- R2 NO_FABRICATION：所有节点存在性/连接性判断必须来自文件读取
- R1 EVIDENCE_FIRST：每个 ❌/⚠️ 必须有具体文件路径证据
- R3 READ_FIRST：Step 1-4 的所有判断必须基于真实读取

推荐注入：
- R6 ARTIFACT_FIRST：生成正式健康报告
```

## 三、人层 vs Agent 层张力分析

```
人层「不言而喻」的假设：
- 「场景发现」结果与人的直觉一致（比如人知道「产品开发是一个场景」）
  AI 可能归纳出不同粒度的场景分组

- 「大闭环」包含认知节点是不言而喻的
  Agent 层需要强制 2c/4c/4d 检查，不能假设 AI 会自动想到

AI 可能产生歧义的点：
- Step 1d「按工作类型聚类」没有精确算法 → 加注：按「role 的触发场景描述中的动词/领域词」聚类
- Step 4b「有效输入」的判断主观性较强 → 加注：检查 Step N+1 的前置条件是否与 Step N 的输出格式匹配

两层对齐后的修订：
- Step 1d 补充：以 DOMAIN-REGISTRY 已有域名称作为聚类参考，但不强制对齐
- Step 4 补充：验证时「节点存在」必须基于 SKILL-INDEX 的实际行记录
```

## 四、复杂度评估

```
与现有 Skill 的交互点：
- skill-closure-verifier-meta：功能互补（节点级 vs 场景级），触发词区分清晰
- skill-system-health-check：功能互补（Loop 1 内部一致性 vs 全场景），无重叠
- DOMAIN-REGISTRY.md：读取参考，不修改
- 所有 role-* SKILL.md：Step 2 读取，不修改

最终级别：Level 4（跨所有工作域，动态发现，多文件读取，报告编排）
需走 skill-system-evolution-executor Phase 0~5
```
