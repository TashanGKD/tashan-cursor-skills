## 一、人层设计（Human Layer）

### 1.1 解决什么问题？

```
有了多个视角的提示词后，AI 自动按规范批量调用图像 API、生成候选图、
以统一命名规范保存，并输出可以被审核 Skill 直接消费的生成清单。
避免每次手动调 API、随意命名、文件散落各处。
```

### 1.2 触发条件（人的视角）

```
典型触发语句：
- 「生成候选图」
- 「按提示词批量生图」
- 「开始生图」
- 「image-ideation 做完了，接着生图」

触发场景：
- image-ideation 完成后，用户确认提示词方向，准备生成实际图片
- 用户已有现成提示词（跳过 image-ideation），直接要生图

明确不应该触发的情况：
- 用户还在改提示词、没有确认视角 → 继续停在 image-ideation 阶段
- 用户只是想看看提示词效果（不保存文件）→ 可以临时生成但不走本 Skill 的完整存档流程
```

### 1.3 成功标准（人的视角）

```
产出物：
- N 张候选图（N = ideation 文件中有效视角数量）
- generation-log-[位置标识]-v[N].md（生成清单，含：文件名/视角/提示词/模型/时间戳）
- images 目录下规范存放的图片文件

质量判断标准：
- 每张图有对应的视角标注（不是无名图片）
- 文件命名包含足够信息（文章名/位置/版本/视角/模型）
- generation-log 完整，可以追溯任何一张图的来源提示词
- 失败的图（API 超时/质量极差）有记录和重试机制
```

### 1.4 MVP 边界

```
✅ 做：
- 读取 ideation 文件中的提示词（或用户直接提供的提示词）
- 按顺序调用 Qwen Image API（qwen-vl-max 或 qwen-image-2.0-pro）
- 按规范命名保存图片（见命名规范）
- 输出 generation-log MD 文件
- 失败重试（最多 2 次）

❌ 不做：
- 不评估图的质量（那是 image-reviewer 的事）
- 不选图（那是用户 + image-reviewer 的事）
- 不修改提示词（那是 image-ideation 的事）
- 不生成数据图表（R/Python/Echarts 等代码生成的图）
```

## 二、Agent 层设计（Agent Layer）

### 2.1 触发条件（AI 视角）

```
触发条件（精确描述）：
IF 用户说「生成候选图」/ 「开始生图」/ 「按提示词生图」
   AND 存在 ideation 文件 OR 用户在对话中提供了完整提示词
THEN 触发本 Skill

与其他组件的优先级：
- 必须在 image-ideation 之后（或用户确认有提示词后）才能触发
- 若用户说「生图」但没有提示词来源 → 先触发 image-ideation
- 若用户说「换个风格重生成」→ 读取当前视角的提示词修改风格后重生成，
  不重新做 image-ideation
```

### 2.2 执行步骤（精确序列）

```
Step 1  读取提示词来源
        输入：ideation 文件路径（优先）或对话中用户提供的提示词
        操作：解析 ideation 文件，提取所有「有效视角」的完整提示词
              若用户指定了只生成某几个视角，按用户指定范围
        输出：[内部] 提示词列表（含：视角名、提示词内容）
        失败时：若找不到 ideation 文件，提示用户确认路径或直接粘贴提示词

Step 2  确认生成参数
        输入：提示词列表
        操作：确认以下参数（若已有默认值则静默使用）：
              - 模型：默认 qwen-image-2.0-plus（或最新可用 Qwen 图像模型）
              - 图片尺寸：默认 1024×1024（正方形），可选横版 1792×1024
              - 数量：每个视角生成 1 张（MVP；后续可扩展为 2 张 A/B 变体）
        输出：[内部] 生成参数配置
        失败时：若模型不可用，自动降级到下一个可用模型并告知用户

Step 3  按视角顺序调用 Qwen Image API
        输入：每个视角的提示词 + 生成参数
        操作（对每个视角循环执行）：
          a. 调用 Qwen Image API（POST /images/generations）
          b. 等待响应（超时设置：60秒）
          c. 成功 → 下载图片数据
          d. 失败 → 记录错误，重试 1 次；再失败 → 跳过并标记「[生成失败]」
        输出：图片数据（内存/临时文件）
        失败时：若超过 50% 的图片生成失败，暂停并提示用户检查 API 状态

Step 4  按规范命名并保存图片
        输入：图片数据 + 视角名 + 文章名/位置信息
        操作：按以下命名规范保存：
              [文章名缩写]_[位置标识]_v[版本号]_[视角缩写]_[模型缩写].png
              示例：战略框架_内外循环_v1_流程视角_qwen.png
              保存路径：[文章目录]/images/candidates/
        输出：图片文件（PNG 格式）
        失败时：若目录不存在，自动创建；若文件名重复，在后面加 _b

Step 5  生成 generation-log 文件
        输入：全部生成结果（成功 + 失败）
        操作：按以下格式写入文件：
              文件路径：[文章目录]/images/candidates/
                        generation-log-[位置标识]-v[版本号].md
        输出：generation-log MD 文件（见格式规范）
        失败时：若写文件失败，在对话中输出并提示用户手动保存

Step 6  告知用户并衔接下一步
        输出：「✅ 生成完成：共 N 张图（M 成功 / K 失败）
               保存路径：[目录]
               generation-log：[路径]
               说「审核候选图」→ 触发 image-reviewer（推荐）
               或直接打开目录查看」
```

### generation-log 文件格式规范（附录）

```markdown
# 生成清单：[文章名] - [位置描述]

> 版本：v[N] | 日期：YYYY-MM-DD HH:MM | 模型：[模型名] | 来源：[ideation 文件路径]

| 视角名 | 文件名 | 状态 | 提示词摘要 |
|--------|--------|------|-----------|
| 流程视角 | 战略框架_内外循环_v1_流程视角_qwen.png | ✅ 成功 | 双层循环结构，内圈...外圈... |
| 对比视角 | 战略框架_内外循环_v1_对比视角_qwen.png | ✅ 成功 | 左旧路径/右新路径对比... |
| 隐喻视角 | 战略框架_内外循环_v1_隐喻视角_qwen.png | ❌ 生成失败（超时）| 飞轮+螺旋... |

## 提示词完整记录

### 视角一：流程视角
[完整提示词]

### 视角二：对比视角
[完整提示词]
```

### 2.3 边界条件

```
情况A：提示词中包含违禁内容（API 拒绝）
       → 记录错误 + 提示用户修改提示词的哪个部分（根据 API 错误信息）

情况B：API key 失效或余额不足
       → 立即停止，告知用户：「API 认证失败，请检查 [key 来源位置] 的有效性」
         不继续重试（避免浪费）

情况C：用户想生成比 ideation 视角更多的图（自己临时加提示词）
       → 接受，将临时提示词标记为「[临时视角-用户自定义]」，同样写入 generation-log

情况D：同位置已有旧版 generation-log（v1/v2...）
       → 自动命名为下一个版本号，不覆盖（历史留痕原则）
```

### 2.4 依赖声明

```
依赖的其他 Skill/Agent/Rule：
- image-ideation：消费其产出的 ideation 文件；若无此文件则需用户直接提供提示词
- Qwen Image API：Step 3 核心依赖；API key 存储位置：[用户在环境变量或配置文件中提供]
- article-image-styles：风格库编号在提示词中已由 image-ideation 指定，本 Skill 不额外读取

会影响的其他 Skill/Agent/Rule：
- image-reviewer：消费本 Skill 产出的 generation-log + 图片文件
```

### 2.5 常见失败模式

```
失败模式1：图片保存成功但命名不规范（随机文件名）
  → 根本原因：Step 4 跳过了命名规范，直接用 API 返回的默认文件名
  → 预防方法：Step 4 强制执行命名规范，API 返回文件名不可直接使用

失败模式2：generation-log 里提示词只有摘要，不是完整原文
  → 根本原因：AI 为了简洁，在 log 里只写了提示词片段
  → 预防方法：Step 5 明确要求「提示词完整记录」节必须包含全文

失败模式3：部分视角失败但 AI 没有告知，用户误以为全部成功
  → 根本原因：Step 6 的告知没有区分成功/失败数量
  → 预防方法：Step 6 格式强制：「M 成功 / K 失败」，失败数>0 时必须列出失败视角

回退动作：
- 若 API 持续失败 → 输出提示词清单，建议用户手动在 Qwen 界面生成后提供图片路径
- 若文件保存失败 → 在对话中告知图片 URL，用户手动下载
```

### 2.6 强绑定 Rule

```
强绑定（必须遵守）：
- R2 NO_FABRICATION：不能捏造「图已生成」；API 未成功返回时不得声称生成完成
- R9 VERSION_CONTINUITY：generation-log 版本号必须连续，不可跳号或覆盖旧版

推荐注入：
- R6 ARTIFACT_FIRST：每次生成必须有 generation-log 文件，不接受只在对话里说「图生成好了」
```

## 三、人层 vs Agent 层张力分析

```
人层设计中「不言而喻」的假设：
- 人觉得「批量生图」很直接——按顺序生成就好
  → AI 需要知道：每张图生成后必须立即保存 + 记录，不能等全部生成完再处理
    （防止 API 部分失败导致数据丢失）
- 人觉得「命名规范」很显然
  → AI 需要被明确告知每个字段的来源和格式（文章名缩写如何截取？版本号如何递增？）

AI 可能产生歧义的点：
- 「批量」可能被理解为并发请求 → 应当是顺序请求（Qwen API 有并发限制）
- generation-log 的「提示词摘要」vs「完整提示词」 → 已明确要求完整原文
```
