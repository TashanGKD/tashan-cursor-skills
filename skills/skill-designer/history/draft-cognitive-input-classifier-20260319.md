## 一、人层设计

### 1.1 解决什么问题？
```
任何进入系统的输入没有统一的「先分类、再路由」入口。AI 目前直接执行
最像当前上下文的操作，导致：
① 应更新认知体系的输入被当作任务执行（跳过认知整合）
② 应执行任务的输入在认知讨论中耽误

cognitive-input-classifier 作为「路由前置层」：输入到来 → 先分类 →
明确走路径A（认知更新）还是路径B（任务执行）→ 再触发对应 Skill
```

### 1.2 触发条件（用户视角）
```
典型触发：
- 用户输入意图不明确（既不含明确任务动词，也不含明确认知记录词）
- 系统自检时发现输入没有被正确路由
- 用户明确说「帮我判断这是认知更新还是任务执行」

明确不应触发：
- 输入意图已清晰（「帮我写代码」→ 直接执行）
- 用户已明确说「这是个任务」或「我想记录一个认知」
```

### 1.3 成功标准
```
产出物：路由决策结论（对话输出）
质量标准：
- 明确输出「路径A / 路径B / AB混合」，不模糊
- 给出判断理由（1句话）
- 混合输入时拆分为认知部分和任务部分，分别路由
- 用户看到结论后能立刻知道下一步触发什么 Skill
```

### 1.4 MVP 边界
```
✅ 做：
- 接受任意输入，输出路由结论（A/B/AB混合）+ 判断理由
- 混合输入拆分为认知部分和任务部分，分别路由
- 无法判断时追问 1 次，仍不确定则默认路径B

❌ 不做：
- 执行路由后的操作（由对应 Skill 负责）
- 记录历史路由决策
- 自动触发下游（只输出结论，不触发，等用户确认）
```

## 二、Agent 层设计

### 2.1 触发条件（精确描述）
```
触发条件：
  IF 用户输入满足以下任一：
    - 意图标签不清晰（不含明确任务动词，也不含明确认知标记词）
    - 用户说「帮我判断这是什么类型的输入」
    - 上游 Skill 无法确定应走哪条路径
  THEN 触发 cognitive-input-classifier

优先级：
- vs cognitive-capture-fragment：本Skill先于后者运行；后者在输入已被确认为「认知碎片」后触发
- vs 任务Skill：本Skill先于任务Skill运行；任务Skill在输入已被确认为任务后触发
- 用户意图已经非常清晰时，不应触发本Skill（直接触发对应Skill）
```

### 2.2 执行步骤
```
Step 1  分析输入的意图信号
        检查以下信号词（基于实际输入文本，不猜测）：
        
        认知信号（路径A指向）：
          「我发现」「我觉得」「我想到」「记录一下」「碎片」「理解X」
          「X的规律是」「这让我想到」「有个洞见」「我注意到」
        
        任务信号（路径B指向）：
          「帮我做」「写X」「执行」「实现」「修复」「部署」
          祈使句动词开头（「新建」「设计」「分析」「整合」等带具体产物的动词）
        
Step 2  输出路由结论（格式固定，不允许模糊）
        
        路径A（认知更新）：
          「📗 路径A：认知更新。
            建议触发：cognitive-capture-fragment
            判断理由：[引用输入中的具体信号词，1句话]」
        
        路径B（任务执行）：
          「📋 路径B：任务执行。
            建议触发：[基于输入内容推断的对应任务Skill]
            判断理由：[引用输入中的具体信号词，1句话]」
        
        AB混合（同时含两类信号）：
          「🔀 路径AB混合：
            认知部分：[具体内容] → 路径A → cognitive-capture-fragment
            任务部分：[具体内容] → 路径B → [对应任务Skill]
            建议先处理哪个？」
        
        无法判断（无明确信号词）：
          「❓ 无法判断意图。请问：这个输入的核心目的是
            (A) 理解/记录某件事  (B) 做成某件事？」
          追问上限：1次
          若仍无法判断 → 「默认路径B（任务执行），如有误请纠正」

Step 3  等待用户确认，不自动触发下游
        输出结论后停止，等用户说「确认」或「用路径X」再触发对应Skill
```

### 2.3 边界条件
```
情况A：输入很长（>500字），同时含大量认知和任务内容
→ 处理：提取「最核心的1句意图」做判断，不尝试全量分析
→ 提示：「这个输入很长，我基于核心意图判断：[结论]，如有偏差请纠正」

情况B：用户不认可路由结论
→ 处理：接受用户纠正，输出更正后的结论，不辩解
→ 格式：「收到，更正为路径[X]，触发 [对应Skill]」

情况C：完全没有输入内容（空输入）
→ 处理：告知「需要有实际内容才能做路由判断」，不输出结论
```

### 2.4 依赖声明
```
依赖的其他 Skill/文件：无（纯输入分析，不读文件）

会影响的其他 Skill（下游）：
- cognitive-capture-fragment（路径A的主要下游）
- 各任务Skill（路径B的下游，由具体输入内容确定）
```

### 2.5 常见失败模式
```
失败模式1：混合输入被强行归类为单一路径（A or B），忽略了另一类
  → 根本原因：AI 取了「占比更大的那类」当作唯一结论
  → 预防方法：Step 2 在同时发现认知信号和任务信号时，强制走「AB混合」分支

失败模式2：「无法判断」时输出「这是一个既有认知又有任务性质的输入」模糊结论
  → 根本原因：AI 把「无法判断」等同于「给出模糊结论」
  → 预防方法：Step 2 明确：无法判断只有两个出口——追问一次 OR 默认路径B

失败模式3：用输入内容里没有的信号词来支撑判断（捏造理由）
  → 根本原因：AI 基于「感觉」判断，不基于文本证据
  → 预防方法：判断理由必须引用输入中实际存在的词句，不允许推断

回退动作：
- 若用户纠正路由结论 → 接受，输出更正结论，不辩解
- 若输入内容歧义无法解决 → 默认路径B，告知可纠正
```

### 2.6 强绑定 Rule
```
强绑定（必须遵守）：
- R2 NO_FABRICATION：判断理由必须基于输入中实际存在的信号词，不允许推断或捏造

推荐注入：
- R13 RULE_BEATS_STYLE：准确判断优先于表述流畅，宁可输出简短的「路径B」也不输出漂亮的模糊结论
```

## 三、人层 vs Agent 层张力分析
```
人层「不言而喻」的假设：
1. 「意图不明确时触发」——人知道什么是「不明确」，AI 需要信号词列表才能判断
   修订：2.1 改为枚举具体信号词，不用「不明确」的概念

2. 「混合输入时拆分」——人知道如何识别「混合」，AI 需要「同时发现两类信号词」的具体规则
   修订：Step 2 加入「AB混合」的判断条件

AI 可能产生歧义的点：
1. 「基于实际输入文本中的信号词判断」vs「基于上下文语义推断」
   → AI 倾向于语义推断，容易绕过信号词规则直接给结论
   → 修订：2.5 失败模式3 + R2 强绑定 共同预防
```
