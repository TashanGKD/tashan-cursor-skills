## 一、人层设计（Human Layer）

### 1.1 解决什么问题？

```
郑总写技术分析文章、产品思考长文、对外分享时，没有结构化入口——
只有 wechat-article-writer（强绑定微信模板/HTML），写通用长文时无法调用。
这导致通用文章写作无 WG 风格约束、无大纲确认、无审稿闭环。
```

### 1.2 触发条件（人的视角）

```
典型触发语句：
- 「帮我写一篇技术文章」
- 「写一篇对外分享」
- 「写一篇产品分析/产品思考长文」
- 「帮我写一篇通用文章/长文/文章草稿」
- 「写一篇个人随笔/思考」

触发场景：
- 郑总要写一篇非公众号的长文（技术博客/产品分析/内部分享/学术周边写作）
- 需要 WG 风格约束但不需要微信排版

明确不应该触发的情况：
- 用户说「写一篇公众号文章/他山文章/推文」→ wechat-article-writer
- 用户说「写 SI/补充材料/LaTeX」→ academic-si-writer
- 用户问「如何写作」→ 只是问答，不触发
```

### 1.3 成功标准（人的视角）

```
产出物：完整 Markdown 文章草稿（含标题、目录、正文）
质量判断标准：
  - 风格符合 WG 手册（换位思考、结论先行、无AI腔）
  - 大纲已经过用户确认后才写正文
  - 自动完成 article-proofreading 审稿
  - 结构清晰，可直接用于对外分享或进一步加工
```

### 1.4 MVP 边界

```
✅ 做：
  - Step 0 读 WG，提取风格约束
  - 4种文章类型（技术分析/产品思考/对外分享/个人随笔）
  - 大纲确认后写正文
  - F-022 挑战者自检
  - 触发 article-proofreading 审稿

❌ 不做：
  - HTML 转换（WeChat 专用，不在本 Skill 范围）
  - 排版核查脚本（check_html.py 只适用于 WeChat HTML）
  - 配图生成（可选，用户可手动调用 article-image-angles）
  - 学术格式/LaTeX（→ academic-si-writer）
```

## 二、Agent 层设计（Agent Layer）

### 2.1 触发条件（AI 视角）

```
触发条件（精确描述）：
  IF 用户输入包含「写一篇[技术/产品/通用/对外/分享/长文/随笔/分析]文章」
  AND 输入中不含「公众号/推文/他山文章/微信/WeChat」
  THEN → 触发本 Skill

  IF 不确定是否是公众号 → 先问：「这篇文章是要发公众号吗？」
    - 是 → 切换 wechat-article-writer
    - 否 → 继续本 Skill

与其他组件的优先级：
  - 若同时匹配 wechat-article-writer 触发词 → wechat-article-writer 优先
  - 若包含「SI/补充材料/LaTeX」→ academic-si-writer 优先
  - 若只是询问写作方法 → cognitive-ask 或直接回答，不触发本 Skill
```

### 2.2 执行步骤（精确序列）

```
Step 0  读取写作习惯与风格手册
        Read: _内部总控/认知结构/L1_系统性文档/待建维度/写作习惯与风格手册.md
        提取：核心写作哲学 + 语言风格禁忌 + 标题写法规范 + 信息密度原则
        失败时：告知用户 WG 文件不存在，但继续执行（退化到 AI 默认风格）

Step 1  确认文章类型与目标
        向用户提问（如未说明）：
        「这篇文章属于哪种类型？
          A. 技术分析——技术方案/框架对比/系统设计
          B. 产品思考——产品设计/商业分析/战略判断
          C. 对外分享——演讲稿/内部培训/面向读者的解释性文章
          D. 个人随笔——思考记录/反思/实验性写作」
        同时确认：目标读者是谁 + 预期长度（500/1000/2000/3000字以上）
        输出：文章类型 + 读者画像 + 长度目标（存入工作记忆）
        失败时：给出推荐类型，等用户确认后继续

Step 2  生成文章大纲（等用户确认）
        基于 Step 0 WG 的「换位思考：从读者决策链倒推结构」
        产出：
          - 文章标题（遵循 WG 标题规范：结论型/问题型，避免话题型）
          - 3-6 个章节标题
          - 每章核心论点一句话
        ⚠️ 必须等用户确认大纲后，才能进入 Step 3
        失败时：提供 2-3 个可选大纲方向

Step 3  写正文
        遵循 Step 0 WG 约束：换位思考 + 结论先行 + 无AI腔 + 信息密度
        文章结构约定（通用模板）：
          - 标题
          - [可选] 摘要（对外分享类/长文必须）
          - 正文章节（按确认的大纲）
          - [可选] 结语（长文必须，结语需覆盖全文主题）
        输出格式：Markdown，写入对话或建议保存路径
        失败时：若某章节缺乏素材，明确告知并跳过，不编造

Step 4  F-022 全节点挑战者反思
        以「第一次读这篇文章的陌生读者」视角执行3条挑战：
        1. 理解障碍：哪一段读者在没有背景知识时会卡住？
        2. 标题-正文一致性：标题承诺的核心信息，正文是否确实兑现？
        3. 最弱段落：哪一段价值最低？删掉后文章是否更好？
        若发现可修复问题 → 修复后进 Step 5
        若无重大问题 → 输出「内容自检：[轻微问题/可接受弱点]」

Step 5  触发 article-proofreading
        将写好的文章传入 article-proofreading skill
        检查：AI腔/标题规范/绝对表达/结语完整性
        自动执行，不需要用户再次确认
```

### 2.3 边界条件

```
情况A：用户只提供部分草稿 → 进入补全模式（Step 1 识别现有类型和结构，Step 2 识别缺失部分，Step 3 补全）
情况B：WG 文件不存在/读取失败 → 告知用户，继续执行（退化到 AI 训练风格）
情况C：用户打断写作 → Step 3 输出已写的部分，用户重新激活时从上次停止位置继续
情况D：不确定是否发公众号 → Step 1 前询问确认
```

### 2.4 依赖声明

```
依赖的其他 Skill：
- article-proofreading：Step 5 调用，传入完成的文章

不强制但可选调用：
- article-image-angles：用户要求配图时可单独调用

会被影响的 Skill：
- wechat-article-writer：共享 article-proofreading，触发词须明确区分
```

### 2.5 常见失败模式

```
失败模式1：跳过大纲确认直接写正文
  根本原因：AI 认为大纲「显然」直接能写
  预防方法：Step 2 末尾明确「⚠️ 必须等用户确认大纲后，才能进入 Step 3」

失败模式2：漏读 WG，直接按训练知识写
  根本原因：Step 0 被视为可选步骤
  预防方法：Step 0 放在第一位，描述明确「写稿前必做」

失败模式3：文章类型判断错误（如把对外分享写成技术文档风格）
  根本原因：4种类型区分不清
  预防方法：Step 1 给出每种类型的典型示例

失败模式4：结语缺失（长文）
  根本原因：AI 在所有内容写完后认为「已完成」
  预防方法：Step 3 明确「长文必须有结语」

回退动作：
- 若用户确认大纲后要求修改 → 回到 Step 2 调整大纲
- 若 Step 5 审稿发现严重问题 → 回到对应章节修改，不直接输出
```

### 2.6 强绑定 Rule

```
强绑定：
- R2 NO_FABRICATION：不编造内容、案例、数据

推荐注入：
- R6 ARTIFACT_FIRST：输出应为完整 Markdown 文档
- R13 RULE_BEATS_STYLE：正确性优先于「写得流畅」
```

## 三、人层 vs Agent 层张力分析

```
人层隐含假设 → Agent 层需显式告知：
- 「写文章」隐含「先确认大纲」→ 已在 Step 2 加 ⚠️ 强制确认
- 「通用」很模糊，郑总心里有4种类型 → 已在 Step 1 列出并要求选择
- 「WG 风格」意味着读最新版本，不是套用旧摘要 → Step 0 强制 Read
- 结语对「长文」是必须的，对短文是可选的 → Step 3 明确区分

AI 可能产生歧义的点：
- 「对外分享」可能被理解为要发到公众号 → Step 1 前询问渠道
- 「大纲确认」可能被理解为「给个大纲，不等确认」→ Step 2 加⚠️

两层对齐后修订：
- Step 1 增加「发布渠道确认」问题
- Step 2 增加强制等待用户确认的约束
```

```
- wechat-article-writer：触发词可能重叠（「写一篇文章」），需要优先级规则
- article-proofreading：被本 Skill 在 Step 5 调用

Level 判断：
[x] Level 2：新建独立组件
    触发词与 wechat-article-writer 部分重叠，但可通过「不含公众号/微信」的负条件消歧
    → 不升 Level 3，因为无新增依赖（article-proofreading 本来就存在，只是被调用）

最终级别：Level 2
```
