# Skill 产品定义卡片：image-ideation（图示构思）

> 系统位置：绘图三件套 Skill 1/3，被写作者角色调用，产出传入 image-generator
> 关系类型：triggered-by（wechat-article-writer / 任意写作场景）/ triggers（image-generator）

## 一、人层设计（Human Layer）

### 1.1 解决什么问题？

```
写作时遇到「这里需要一张图」，AI 不再直接生成，
而是先做多视角构思 + 沙盘推演，确保每张候选图背后都有清晰的叙事假设，
而不是随机地「配一张好看的图」。
```

### 1.2 触发条件（人的视角）

```
典型触发语句：
- 「这段需要配图，帮我想想」
- 「这里想要一张图，先分析一下」
- 「帮我做图示构思」
- 「这段文字背后可以画什么图？」
- （写作者角色内部）遇到段落中标记 <!-- 需要配图 --> 时自动触发

触发场景：
- 写作者正在写某一段内容，认为有图比纯文字更能揭示关系
- 用户已有一段写好的文字，想为其配图但不知道画什么
- 现有图觉得不好，想重新构思

明确不应该触发的情况：
- 用户已经有明确提示词了，只需要生图 → 直接触发 image-generator
- 用户想要修改已有图的细节（颜色/风格）→ 直接调整提示词重生成
- 用户想看风格变体（不换视角，只换画风）→ 触发 article-image-styles
```

### 1.3 成功标准（人的视角）

```
产出物：ideation-[文章名]-[位置标识].md
  - 包含 2-4 个独立视角
  - 每个视角有：视角名称 / 叙事假设 / 目标读者维度 / 完整提示词 / 挑战者反思
  
质量判断标准：
- 不同视角之间有实质性差异（不是同一张图的风格变体）
- 每个视角的提示词对应一个不同的「关系/结构」揭示
- 挑战者反思明确指出了每个视角的局限或适用条件
- 看完构思文件，用户能清晰感受到「不同视角下图会长什么样」
```

### 1.4 MVP 边界

```
✅ 做：
- 分析内容背后的关系类型（因果/层级/流程/对比/循环/分布）
- 从 2-4 个维度生成独立视角（叙事视角 + 读者维度）
- 每个视角写完整提示词（构图 + 风格 + 颜色 + 禁止元素）
- 每个视角做挑战者反思（揭示关系？还是文字装饰？）
- 输出结构化的 ideation MD 文件
- 参考全场景绘图指南知识库选择合适图类型

❌ 不做：
- 不实际调用 API 生成图片（那是 image-generator 的事）
- 不做风格变体（同一视角的多种画风）→ article-image-styles
- 不处理数据类图表（R/Python 生成的统计图）→ 另立 Skill
- 不做审核和选图 → image-reviewer
```

## 二、Agent 层设计（Agent Layer）

### 2.1 触发条件（AI 视角）

```
触发条件（精确描述）：
IF 用户说「这里需要图/配图/帮我想想图/图示构思」
   OR 写作者角色遇到段落中 <!-- 需要配图 --> 标记
   AND 用户没有提供现成的提示词
THEN 触发本 Skill

与其他组件的优先级：
- vs article-image-angles：本 Skill 更通用（不限公众号）且输出为可保存的文件；
  如果触发词包含「公众号」且用户只想要提示词对话输出（不需要存文件），
  article-image-angles 优先；否则本 Skill 优先
- vs image-generator：本 Skill 在前，image-generator 在后；
  如果用户已有明确提示词，跳过本 Skill 直接到 image-generator
```

### 2.2 执行步骤（精确序列）

```
Step 1  读取内容上下文
        输入：当前段落/章节文字（从对话上下文或用户粘贴）
              + 文章标题和整体主题（如可获取）
              + 全场景绘图指南（关系类型 → 图类型映射）
        操作：理解这段文字的核心论点和想要揭示的「关系」
        输出：[内部] 关系类型标签（如：「循环因果」「层级演变」「对比张力」）
        失败时：若无法识别段落核心关系，向用户追问一个问题：
               「这段文字最想让读者记住的一个结构/关系是什么？」

Step 2  生成 2-4 个独立叙事视角
        输入：Step 1 识别的关系类型 + 文章上下文
        操作：从以下维度各取 1 个视角构建候选：
              - 数据/量化角度（用图表/数字表达核心论点）
              - 隐喻/类比角度（用视觉隐喻让抽象概念具象化）
              - 流程/机制角度（展示系统如何运作）
              - 对比/张力角度（突出转变/差异/矛盾）
              - 场景/情境角度（还原真实语境）
              注：不是每次都用5个，选最相关的2-4个
        输出：[内部] 视角清单（视角名 + 一句话叙事假设）
        失败时：若只能想出1个有实质性差异的视角，如实告知并说明原因

Step 3  沙盘推演（核心步骤，不可跳过）
        输入：每个视角的叙事假设
        操作：对每个视角，AI 在内部完成以下推演：
              「如果按这个视角生成图，读者会看到什么？
               他们脑子里会形成什么形象（spatial model）？
               这个形象揭示了文字里说的关系吗？
               还是说，只是把文字里的词汇列在图上？」
        输出：[内部] 每个视角的沙盘推演结论（「有效揭示」/「文字映射」/「部分揭示」）
        失败时：若某视角推演结论为「纯文字映射」，
               修改该视角叙事假设，或标注「[低质量视角，建议跳过]」

Step 4  为每个有效视角写完整提示词
        输入：每个视角的叙事假设 + 沙盘推演结论
        操作：为每个「有效揭示」或「部分揭示」的视角写提示词，格式：
              - 构图描述（元素 + 布局 + 层次关系）
              - 风格描述（从图片风格库选，或指定）
              - 颜色基调（协调他山品牌色蓝绿 rgb(111,167,170)；或中性学术风）
              - 禁止元素（「无文字/无人脸/无版权图案」为标配）
        输出：每个视角的完整提示词（中文，200-400字）
        失败时：若写不出无歧义的构图描述，先输出草稿并标注「[需用户补充：XX]」

Step 5  挑战者反思（每视角一条，不可跳过）
        输入：每个视角的提示词
        操作：从「挑剔读者」视角逐一评估：
              「这个提示词生成的图，去掉它文章还完整吗？
               还是图有额外价值，让读者理解了文字没说完的部分？」
              同时标注：适合什么类型读者？不适合谁？
        输出：每个视角的挑战者评论（2-3句，诚实指出局限）
        失败时：若所有视角都通不过挑战者检验，告知用户并建议是否需要这张图

Step 6  输出并保存 ideation 文件
        输入：Step 2-5 的全部产出
        操作：按以下格式写入文件：
              文件路径：[文章所在目录]/images/ideation/
                        ideation-[位置标识]-v1.md
              格式：见「ideation 文件格式规范」（本卡片附录）
        输出：ideation-[位置标识]-v1.md
        失败时：若文章目录不存在，先提示用户或创建标准目录结构

Step 7  告知用户并等待确认
        输出：「✅ 图示构思完成，已保存至 [路径]
               共 N 个视角，其中推荐优先考虑：[视角X]（理由：[一句话]）
               说「生成候选图」→ 触发 image-generator
               或直接告知要修改哪个视角」
```

### ideation 文件格式规范（附录）

```markdown
# 图示构思：[文章名] - [位置描述]

> 版本：v1 | 日期：YYYY-MM-DD | 对应段落：[段落首句]
> 内容关系类型：[因果/层级/循环/对比/分布/流程]

---

## 视角一：[视角名称]

**叙事假设**：[这个视角想让读者看到什么关系]
**适合读者**：[谁会从这个视角获益最多]
**沙盘推演**：[想象生成结果后的自我评估]

**提示词**：
[完整提示词，200-400字]

**挑战者反思**：[诚实评估这个视角的局限，适用和不适用场景]

---

## 视角二：[视角名称]
...

---

## 推荐优先视角

| 视角 | 推荐指数 | 理由 |
|------|---------|------|
| 视角一 | ★★★★☆ | ... |
| 视角二 | ★★★☆☆ | ... |
```

### 2.3 边界条件

```
情况A：用户提供的段落没有明确的「关系」可视化
       → 告知：「这段文字主要是[叙述/列举/引用]，可能不需要配关系型图；
         如果需要图，建议用[场景图/装饰图]，触发 article-image-angles 更合适」

情况B：用户想要纯装饰性图片（没有信息价值，只是视觉吸引）
       → 提示：「你需要的是装饰性配图，而非信息图示；
         建议用 article-image-angles + article-image-styles 处理」

情况C：段落内有多个可以分别配图的子概念
       → 询问：「这段有A/B/C三个可以独立配图的地方，是分别构思还是选一个最核心的？」

情况D：已存在同位置的旧版 ideation 文件
       → 自动命名为 v2/v3，不覆盖旧版（历史留痕）
```

### 2.4 依赖声明

```
依赖的其他 Skill/Agent/Rule：
- 全场景绘图指南.md：Step 1 中识别关系类型时参考；Step 2 选视角时参考图类型矩阵
- article-image-styles：Step 4 写提示词时引用风格库编号
- knowledge-integrity-rules.mdc (R2/R3)：Step 3 沙盘推演时不捏造视觉描述

会影响的其他 Skill/Agent/Rule：
- image-generator：消费本 Skill 产出的 ideation 文件
- wechat-article-writer：本 Skill 被其 Step 3（配图）调用
```

### 2.5 常见失败模式

```
失败模式1：所有视角都是「文字映射」（把段落文字画成方框+箭头）
  → 根本原因：Step 2 没有真正从不同叙事角度思考，只是换了词汇
  → 预防方法：Step 3 沙盘推演强制执行，推演结论为「文字映射」时强制修改

失败模式2：多个视角其实是同一张图的风格变体
  → 根本原因：没有检查各视角的「叙事假设」是否有实质差异
  → 预防方法：Step 2 完成后检查：各视角的叙事假设之间「关系揭示的维度不同吗？」
              若两个视角只在画风/颜色上有差异，合并为一个并用 article-image-styles 处理变体

失败模式3：AI 在 Step 7 推荐了「推荐优先视角」但没有给理由
  → 根本原因：AI 生成推荐但跳过了理由输出
  → 预防方法：Step 7 的推荐格式强制要求「理由：[一句话]」，空理由视为无效输出

回退动作：
- 若 Step 3 沙盘推演全部不通过 → 回退 Step 2，尝试重新生成视角
- 若用户对所有视角都不满意 → 升级：让用户描述「你希望读者看完这张图后，脑子里有什么形象？」
```

### 2.6 强绑定 Rule

```
强绑定（必须遵守）：
- R2 NO_FABRICATION：沙盘推演不能描述「我觉得图会长这样」——只能推断，且必须标注为推断
- R3 READ_FIRST：Step 1 必须先真正读完段落，不能凭标题猜测内容
- R13 RULE_BEATS_STYLE：沙盘推演的诚实结论（「这只是文字映射」）比「给用户一个好看的答案」更重要

推荐注入：
- R6 ARTIFACT_FIRST：每次构思必须输出文件（ideation MD），不接受只在对话里口头给提示词
```

## 三、人层 vs Agent 层张力分析

```
人层设计中「不言而喻」的假设：
- 人觉得「沙盘推演」很自然——想象一下图会长什么样
  → AI 需要被明确告知：推演不是「描述我希望图长什么样」，
    而是「这个提示词真的会让 AI 画出揭示关系的图，还是只是装饰？」
- 人觉得「不同视角」很直观
  → AI 需要被明确告知：「视角」的差异必须体现在「揭示的关系维度不同」，
    不是颜色/风格/构图细节不同

AI 可能产生歧义的点：
- 「沙盘推演」可能被理解为「写更多文字描述」而不是「自我质疑提示词」
  → 已在 Step 3 中明确：推演是以问题形式进行的自我质问
- 「挑战者反思」可能被 AI 当作「称赞再加一条小建议」
  → 已在 Step 5 强调「诚实指出局限」，不是表扬

两层对齐后的修订：
- Step 3 加了具体问题清单（「形象是什么」「揭示了关系吗」「还是文字列表的视觉化」）
- Step 5 加了「去掉这张图文章还完整吗？」作为检验标准
```

## 四、复杂度评估

```
与现有 Skill/Rule 的交互点：
- article-image-angles：功能重叠（都做视角分析），本 Skill 范围更广（不限公众号）
  且产出文件（ideation MD）而非只在对话输出；需要在 role-menu 里明确优先级
- article-image-styles：Step 4 写提示词时依赖风格库；无冲突
- wechat-article-writer：本 Skill 被其 Step 3 调用；需更新 wechat-article-writer 加入触发词路由
- 全场景绘图指南.md：知识库依赖；无冲突

Level 判断依据：
[x] Level 3：新建组件，与多个现有组件有交互

最终级别：Level 3
```
