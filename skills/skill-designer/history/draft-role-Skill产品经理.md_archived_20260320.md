## 一、人层设计（Human Layer）

### 1.1 解决什么问题？

```
skill-designer 将 Skill 产品定义（「应该是什么」）与工程实现（「怎么写」）
捆绑在同一个流程里，在设计复杂 Skill 时，两种认知模式相互干扰。

role-Skill产品经理 让产品定义阶段有独立的专注空间——只聚焦「这个 Skill
应该解决什么问题、对谁有价值、成功标准是什么」，产出高质量产品定义卡片，
再交给 skill-designer 执行工程实现。
```

### 1.2 触发条件（人的视角）

```
典型触发语句：
- 「我有个 Skill 想法，但还没想清楚是什么」
- 「帮我做这个 Skill 的产品定义」
- 「Skill PM，分析一下这个需求」
- 「填写 Skill 产品定义卡片」
- 「这个 Skill 应该是什么，帮我想清楚」

触发场景：
- 用户有 Skill 构想但规格模糊，需要先「想清楚」再实现
- PENDING-SKILLS 中有「待评估」项需要做需求分析
- skill-evolution-planner-meta 输出了演进建议，需要把建议转化为产品规格

明确不应该触发的情况：
- 用户已有清晰规格，直接说「帮我写这个 Skill」→ 触发 skill-designer
- 用户在修改已有 Skill 的具体步骤 → 触发 skill-rule-修改规范
- 「新建Skill」含实现意图 → 触发 skill-designer（内含 PM 功能）
```

### 1.3 成功标准（人的视角）

```
产出物：完整填写的 .cursor/skills/skill-designer/draft-[skill名称].md
（6节全部有内容：人层/Agent层/张力分析/复杂度评估/关卡记录）

质量判断标准：
- 人层：设计者读完能独立判断「这个 Skill 对吗」
- Agent层：AI 执行者读完可以无歧义执行，每步都有失败处理
- 失败模式：至少识别了 2 个常见的执行走错场景
- MVP 边界：明确了「这期做什么，不做什么」
```

### 1.4 MVP 边界

```
✅ 做：
- 双视角产品定义（6节模板，含张力分析和复杂度评估）
- 失败模式追问
- Rule 绑定追问
- 输出 draft-[skill名称].md

❌ 不做：
- 编写 SKILL.md 文件（那是 skill-designer 的工作）
- 运行关卡 A/B/C（那是 skill-designer 的工作）
- 更新 SKILL-INDEX / role-menu（那是 skill-designer 部署步骤）
```

## 二、Agent 层设计（Agent Layer）

### 2.1 触发条件（AI 视角）

```
触发条件（精确描述）：
  IF 用户说了包含「Skill产品定义/Skill PM/帮我把这个Skill想清楚」的语句
  OR 用户描述了一个 Skill 需求但明确表示「先不实现」
  OR PENDING-SKILLS 中有「待评估」项需要处理
  THEN 触发 role-Skill产品经理

与其他组件的优先级：
- vs skill-designer：
  「新建一个Skill」→ skill-designer（含PM功能的完整流程）
  「帮我把这个Skill想清楚，暂不实现」→ role-Skill产品经理
  判断原则：用户是否立刻想实现？是→skill-designer，否→本Skill
```

### 2.2 执行步骤（精确序列）

```
Step 1  读取上下文（R3 READ_FIRST）
        输入：无（读文件）
        操作：
          Read: .cursor/skills/skill-designer/skill-product-definition-template.md
          Read: .cursor/skills/skill-index/SKILL-INDEX.md（获取已有组件列表，防重复）
          Read: .cursor/skills/skill-index/PENDING-SKILLS.md（若从PENDING触发，获取描述）
        输出：已了解当前系统已有组件和本次要定义的 Skill 背景
        失败时：若 template 不存在 → 停止，告知模板缺失

Step 2  确认本次定义目标
        输入：用户指令 or PENDING-SKILLS 条目
        操作：
          若用户已说明 → 确认理解
          若来自 PENDING-SKILLS → 读取该条目描述，复述并确认
          若描述模糊 → 追问：「一句话描述它解决的核心问题？」（只问一个）
        输出：确认的 Skill 名称 + 一句话问题描述
        失败时：若用户无法说清楚核心问题 → 建议先做 issue-tracker 记录再回来

Step 3  人层设计引导（逐节）
        输入：确认的 Skill 概念
        操作（每节结束后切换视角做 Agent 层翻译）：

        3.1 解决什么问题 → 用一句话提炼，确认
        
        3.2 触发条件（用户视角）：
            引导用户说出「在什么场景下会触发这个 Skill」
            追问：「在什么相似但不同的情况下，这些词也可能出现？」（防触发词泛化）
            追问：「什么情况下绝对不应该触发？」
        
        3.3 成功标准：
            「走完这个 Skill 后，用户应该得到什么？」
            「用户如何判断 Skill 工作了？」
        
        3.4 MVP 边界：
            「最小版本是什么？能不能只有这些就能用？」
            「这期不做什么，为什么不做？」
        
        【每节完成后切换视角】：
        「你刚才说的，如果 AI 只看文字会怎么理解？我帮你找出人觉得显然但 AI
        需要被显式告知的部分。」
        
        输出：人层 4 节内容（已和用户确认）
        失败时：若用户跳过某节 → 提示「这节会影响后续 Agent 层规格，是否补充」

Step 4  Agent 层设计引导（逐节）
        输入：人层设计结果
        操作：
        
        2.1 精确触发条件：将人层触发词翻译为 AI 可匹配的语义模式
        2.2 执行步骤：每步包含「输入来源/操作/输出格式/失败处理」
        2.3 边界条件：列出非预期情况的处理方式
        2.4 依赖声明：哪些文件/Skill 需要读取？影响哪些组件？
        
        输出：Agent 层 4 节内容
        失败时：若某步骤输出格式不明确 → 追问「这步的输出应该写到哪里，格式是什么？」

Step 5  失败模式追问
        输入：Step 3-4 的内容
        操作（必问）：
        - 「这个 Skill 最容易在哪里出问题？」
        - 「如果 AI 只走到一半就认为完成了，会在哪个步骤停下？」
        - 「过去有没有类似任务的失败经验？」
        
        输出：2.5 失败模式（至少 2 条）
        失败时：若用户说不知道 → AI 基于步骤推断并提出，请用户确认

Step 6  Rule 绑定追问
        输入：整体设计
        操作（必问）：
        - 「是高风险知识任务吗？」→ R1/R2/R10
        - 「需要正式文件输出吗？」→ R6
        - 「涉及版本管理吗？」→ R9
        
        输出：2.6 强绑定 Rule

Step 7  张力分析
        输入：人层 + Agent 层内容
        操作：对比两层，找出「人觉得显然但 AI 需要被显式告知的隐含假设」
        输出：Section 3 张力分析（至少 2 条）

Step 8  复杂度评估
        输入：依赖声明
        操作：标注 Level 1/2/3/4，说明依据
        输出：Section 4 复杂度评估

Step 9  输出产品定义卡片
        操作：
          Write: .cursor/skills/skill-designer/draft-[skill名称].md
          内容：按模板格式，含所有节的内容
        
        收尾告知：
        「产品定义卡片已保存：draft-[skill名称].md
         若准备好实现，说「实现这个Skill」→ 触发 skill-designer
         skill-designer 在 Step 3 会读取此卡片作为起点，不需要重新定义。」
        
        失败时：若卡片不完整（有节缺失）→ 不允许输出，提示缺失节重新填写
```

### 2.3 边界条件

```
情况A：用户中途想立刻实现
  → 处理：「可以，产品定义卡片写到哪里就先保存，然后触发 skill-designer；
    skill-designer 会从卡片继续，无需重新来过。」

情况B：定义途中发现这其实是两个不同的 Skill
  → 处理：停下，明确说「建议拆成两个 Skill：[A] 和 [B]，分别定义」
    询问用户：「先定义哪个？」

情况C：用户在人层设计时混入技术实现细节
  → 处理：立即提示「这是实现细节，Product Definition 只关注「是什么」，
    不关注「怎么做」；技术选型在 Agent 层规格里写」

情况D：来自 PENDING-SKILLS 的 Skill 描述本身就是清晰的规格
  → 处理：简化 Step 3-4，主要做确认和补充，而非从零引导
```

### 2.4 依赖声明

```
依赖的其他 Skill/Agent/Rule：
- skill-product-definition-template.md：Step 1 读取，是模板来源
- SKILL-INDEX.md：Step 1 读取，避免重复定义已有组件
- PENDING-SKILLS.md：Step 1/2 读取（获取待评估项描述）

会影响的其他 Skill/Agent/Rule：
- skill-designer：本Skill产出的卡片是其 Step 3 的预备输入物
- PENDING-SKILLS.md：完成后更新待评估项状态为「已定义，待实现」
```

### 2.5 常见失败模式

```
失败模式1：人层设计完成后直接跳去讨论实现，未完成 Agent 层翻译
  → 根本原因：工程思维习惯，倾向于「想到就做」
  → 预防方法：Step 3 每节结束后强制执行「切换视角翻译」，不允许跳过

失败模式2：产品定义卡片太宏大，MVP 边界失控
  → 根本原因：没有追问「最小版本」，用户一次想做所有功能
  → 预防方法：Step 3.4 强制追问「只有这些最小功能，是否也能解决核心问题？」

失败模式3：输出了卡片但关键节缺失（如无失败模式、无 Rule 绑定）
  → 根本原因：Step 5/6 被跳过
  → 预防方法：Step 9 写文件前，检查 2.5 和 2.6 两节是否非空；若空则阻断

失败模式4：与 skill-designer 职责边界混淆，开始写 SKILL.md 草稿
  → 根本原因：用户说「帮我想想步骤怎么写」，AI 直接开始写实现
  → 预防方法：收到「步骤/实现/代码」类指令时提示「这是 skill-designer 的范围；
    产品定义阶段我们只确认步骤的存在性和目的，不确认具体写法」

回退动作：
- 若某节不满足质量标准 → 回到该节重新引导，不允许跳过继续
- 若用户强烈要求跳过 → 记录跳过原因，在卡片该节标注「⚠️ 用户选择跳过」
```

### 2.6 强绑定 Rule

```
强绑定（必须遵守）：
- R2 NO_FABRICATION：产品定义中不能捏造「用户会这样使用」的假设，
  所有触发条件必须来自用户真实说法或已验证场景
- R4 DEFINE_BEFORE_USE：使用任何 Skill 相关术语前先定义（如「双视角」含义需确认）

推荐注入：
- R6 ARTIFACT_FIRST：产品定义必须落到文件（draft-*.md），不允许只在对话中口述
- R13 RULE_BEATS_STYLE：卡片的完整性优先于写得漂亮
```

## 三、人层 vs Agent 层张力分析

```
人层设计中「不言而喻」的假设：
1. 「引导用户」——人默认 AI 会问问题，但 AI 需要被告知：哪些问题必问、
   哪些问题可选、什么时候停止追问
2. 「产品定义和实现是分开的」——人清楚边界，但 AI 在对话中可能随时滑向实现
3. 「卡片完整才算完成」——人知道6节都要有，AI 可能在3节后就认为「差不多了」

AI 可能产生歧义的点：
1. 「每节结束后切换视角」——AI 可能理解为「写完人层后再写 Agent 层」（整体切换），
   而非「每一节单独切换」（节内切换）
   修订：Step 3 明确写「每节完成后，立即做该节的 Agent 层翻译，再进入下一节」

2. 「失败时：若用户无法说清楚」→「建议先做 issue-tracker 记录」
   AI 可能忽略这个跳出路径，继续循环追问
   修订：明确说「追问上限：同一问题最多问 2 次，之后给出 AI 的预测版本供用户确认」

两层对齐后的修订（已在 2.2 中体现）：
- Step 3 加入「节内切换」的明确指令
- Step 2 加入「追问上限」约束
```
