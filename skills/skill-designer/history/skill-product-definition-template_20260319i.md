# Skill 产品定义卡片模板

> 使用时机：新建 Skill/Agent/Rule（Level 2+）时，在编写前填写本卡片
> 填写工具：由 skill-designer Skill 引导完成，不建议单独填写
> 文件命名：保存为 `skill-designer/draft-[skill名称].md`，完成后归档到 history/

---

## 基本信息

| 字段 | 内容 |
|---|---|
| 组件名称 | [kebab-case，最多6段] |
| 组件类型 | Skill / Agent（前台/后台）/ Rule（alwaysApply/条件触发）|
| 复杂度级别 | Level 1（补丁）/ Level 2（新增）/ Level 3（集成）/ Level 4（系统）|
| 开发日期 | YYYY-MM-DD |
| 需求来源 | PENDING-SKILLS / CO-BUILD-LOG / 用户直接提出 |

## 一、人层设计（Human Layer）

> 这一层回答「为什么」和「是什么」，让人类设计者/审核者能快速理解和信任这个组件。

### 1.1 解决什么问题？

用一句话描述：这个组件的存在，让 AI 在某类场景下能做到之前做不到的事。

```
[填写一句话描述]
```

### 1.2 触发条件（人的视角）

用户会在什么情况下触发这个组件？用自然语言描述：

```
典型触发语句：
- 「[用户会说的话1]」
- 「[用户会说的话2]」

触发场景：
- [描述用户处于什么状态时需要这个组件]

明确不应该触发的情况：
- [描述容易混淆但不应该触发的场景]
```

### 1.3 成功标准（人的视角）

走完这个组件后，用户应该得到什么？

```
产出物：[文件/报告/状态变更/对话输出]
质量判断标准：[人如何判断「这个 Skill 工作了」]
```

### 1.4 MVP 边界

本版本做什么、不做什么：

```
✅ 做：[最小必要功能]
❌ 不做：[刻意不包含的内容，留给后续版本或其他组件]
```

## 二、Agent 层设计（Agent Layer）

> 这一层回答「怎么做」，让 AI 执行者能精确无歧义地执行。

### 2.1 触发条件（AI 视角）

精确到 AI 需要匹配的语义模式：

```
触发条件（精确描述）：
[用 if-then 逻辑描述：如果用户说了X类语句，且上下文满足Y，则触发]

与其他组件的优先级：
[如果同时满足另一个 Skill 的触发条件，应该选哪个？为什么？]
```

### 2.2 执行步骤（精确序列）

每个步骤必须：有明确的输入来源 + 有明确的输出格式 + 有明确的失败处理

```
Step 1  [步骤名]
        输入：[来自哪里]
        操作：[精确描述，没有歧义]
        输出：[格式和内容]
        失败时：[如何处理]

Step 2  [步骤名]
        ...
```

### 2.3 边界条件

AI 执行过程中可能遇到的非预期情况：

```
情况A：[描述] → 处理方式：[描述]
情况B：[描述] → 处理方式：[描述]
```

### 2.4 依赖声明

```
依赖的其他 Skill/Agent/Rule：
- [名称]：[在哪个步骤中调用，传入什么]

会影响的其他 Skill/Agent/Rule：
- [名称]：[本组件执行后，对它有什么影响]
```

## 三、人层 vs Agent 层张力分析

> skill-designer 填写这一节，帮助发现两层设计之间的潜在矛盾。

```
人层设计中「不言而喻」的假设：
- [列出人类觉得显然但 AI 可能不知道的隐含条件]

AI 可能产生歧义的点：
- [列出步骤中可能被 AI 理解为不同含义的表述]

两层对齐后的修订：
- [发现张力后，对上方人层或 Agent 层的修订说明]
```
