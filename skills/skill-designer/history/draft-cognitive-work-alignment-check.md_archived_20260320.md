## 一、人层设计

### 1.1 解决什么问题？
```
当前健康检查体系（三维：Skill内部一致性/认知内部自洽/域传播完备性）
都在检查「系统内部是否正常」，但没有检查「工作任务是否根植于元思维体系」。

工作任务（产品开发/宣传/运营/认知）可能在「漂浮」——它的设计和执行
与认知根（L1.5原则 or L1文档）没有任何映射关系——却无法被发现。

cognitive-work-alignment-check 提供第四维检查：
任何工作任务 → 能否追溯到 L1.5 原则或 L1 文档中的对应节点？
不能追溯 = 漂浮 = 需要重新锚定或重新评估其必要性。
```

### 1.2 触发条件（用户视角）
```
典型触发：
- 「检查工作任务与认知的对齐」
- 「第四维健康检查」
- 「检查工作任务是否有认知根」
- 「[域名]的工作任务是否在认知结构中有根」
- 「认知体系和工作任务是否自洽」

不应触发：
- 只检查 Skill 内部一致性 → skill-system-health-check
- 只检查认知结构内部自洽 → cognitive-consistency-check
- 只检查域传播完备性 → skill-domain-health-check
```

### 1.3 成功标准
```
产出物：对齐检查报告（写文件）
质量标准：
- 对每个工作域，输出「可追溯 ✅」或「漂浮 ❌」
- ❌ 漂浮时：给出建议的认知锚定方向（对应哪条L1.5原则 or L1文档）
- 整体结论：ALIGNED / PARTIALLY-ALIGNED / UNALIGNED

用户如何判断 Skill 工作了：
- 每个工作域都有明确的结论，不是笼统的「基本对齐」
- ❌ 漂浮的任务有具体的「建议锚定到 [P?] 或 [文档名]」
```

### 1.4 MVP 边界
```
✅ 做：
- 读取工作域清单（DOMAIN-REGISTRY.md 五域）
- 读取元思维体系（L0大脑总地图 + L1.5底层原则库概要）
- 对每个域的核心产出，判断是否能追溯到 L1/L1.5
- 输出对齐报告（写文件）

❌ 不做：
- 自动修复对齐问题（只诊断，不修复）
- 强制要求每个工作步骤都有认知根（只检查「域级别」的核心产出）
- 与 skill-domain-health-check 的结果合并（保持独立报告）
```

## 二、Agent 层设计

### 2.1 触发条件（精确描述）
```
触发条件：
  IF 用户说了包含「认知对齐/工作任务对齐/第四维/认知根」的语句
  OR skill-closure-verifier-meta 请求运行第四维检查
  THEN 触发 cognitive-work-alignment-check

优先级（区分）：
- vs skill-system-health-check：后者检查Skill体系内部一致性，无认知层
- vs cognitive-consistency-check：后者检查认知结构内部自洽，不涉及工作任务
- vs skill-domain-health-check：后者检查域传播完备性，不检查认知根
本Skill检查「工作↔认知」的跨层对齐，是第四维，与前三维互补不重叠
```

### 2.2 执行步骤
```
Step 1  读取检查依据（R3 READ_FIRST）
        Read: _内部总控/skill-system-design/DOMAIN-REGISTRY.md
          → 获取五个工作域及每域的核心产出定义
        Read: _内部总控/认知结构/L0_大脑总地图.md（摘要，了解L1文档列表）
        Read: _内部总控/认知结构/L1.5_底层原则层/底层原则库.md（P1/P2/候选原则）
        
        若任一文件不存在：
          → 停止执行，告知「缺少必要文件，无法执行对齐检查」

Step 2  确认检查范围
        - 默认：检查全部五域（产品开发/认知结构/公司运营/内容宣传/Skill体系）
        - 若用户指定单域：只检查该域

Step 3  对每个工作域执行对齐检查
        对每个域，执行以下判断：
        
        3.1 读取该域在 DOMAIN-REGISTRY 中的「终态输出定义」（该域做完后产出什么）
        
        3.2 追溯检查：该域的核心产出能否在以下任一处找到认知根？
            - L1.5 原则库：P1/P2 or 候选原则（P4?/P5?/P7?/P8?）
            - L1 文档列表（18篇，6个维度）
        
        3.3 判断：
            ✅ 可追溯：能找到对应的L1.5原则 or L1文档，给出追溯路径
            ❌ 漂浮：找不到任何对应，给出建议锚定方向（推荐最可能的L1.5/L1）
            ⚠️ 弱追溯：有对应但关系牵强，标注「需要加强映射」

Step 4  输出对齐报告（R6：必须写文件）
        Write: _内部总控/skill-system-design/cognitive-alignment-report-YYYYMMDD.md
        
        报告格式：
        
        # 认知-工作对齐报告 YYYY-MM-DD
        
        | 工作域 | 核心产出 | 追溯状态 | 追溯到 | 建议行动 |
        |---|---|---|---|---|
        | 产品开发 | [核心产出] | ✅/❌/⚠️ | [L1.5/L1文档] | [建议] |
        
        整体结论：ALIGNED / PARTIALLY-ALIGNED / UNALIGNED
        
        漂浮任务清单（若有）：
        - [域名]：建议锚定到 [P?] 或 [文档名]，因为 [理由]

Step 5  路由后续建议
        ✅ 全部对齐：「认知-工作体系自洽，无需行动」
        ⚠️ 部分对齐：「建议为[域]补充认知映射，触发 cognitive-update-knowledge 加入对应L1文档」
        ❌ 漂浮存在：「建议重新审视[域]的必要性，或触发 cognitive-extract-principle 补充缺失原则」
```

### 2.3 边界条件
```
情况A：某个工作域的认知根「存在但很微弱」（只有一个字词重叠）
→ 处理：标注 ⚠️ 弱追溯，不直接标❌，在报告中给出「需要加强映射」的说明

情况B：用户认为某域不需要认知根（「宣传就是宣传，不需要哲学依据」）
→ 处理：记录用户说明，在报告中标注「用户豁免」，不强制要求

情况C：L1 文档本身就描述了某个工作域（如 [E] 文档体系 对应 公司运营域）
→ 处理：标注 ✅ 直接追溯，这是最强的对齐形式
```

### 2.4 依赖声明
```
依赖的其他 Skill/文件：
- DOMAIN-REGISTRY.md：Step 1 读取（工作域定义）
- L0_大脑总地图.md：Step 1 读取（L1文档列表）
- L1.5_底层原则库.md：Step 1 读取（原则列表）

会影响的其他 Skill（下游）：
- skill-closure-verifier-meta：本Skill的报告可被纳入元验证总报告
- cognitive-update-knowledge：报告建议「补充认知映射」时可触发
- cognitive-extract-principle：报告建议「补充缺失原则」时可触发
```

### 2.5 常见失败模式
```
失败模式1：追溯检查过于宽松，任何相关词都算「已追溯」
  → 根本原因：AI 为了避免输出❌，倾向于找到模糊对应就标✅
  → 预防方法：追溯必须找到「明确命题级别的对应」，不是词语重合

失败模式2：没有读完 L1.5 候选原则（只读了已确认的P1/P2）
  → 根本原因：候选原则在「二、候选原则」区，AI 可能只读了「一、已确认」区
  → 预防方法：Step 1 明确「读取 L1.5 包括已确认原则和候选原则，全部读完」

失败模式3：五域全部标✅（过于乐观），报告无参考价值
  → 根本原因：同失败模式1
  → 预防方法：若全部✅，AI 应自我检查：「有没有追溯过于勉强的域？」至少标注一个⚠️

回退动作：
- 若文件读取不完整 → 标注「基于不完整信息，结论可靠性：中」，不阻断
- 若某域完全找不到认知根 → 不猜测，直接输出❌并给出「建议方向」而非「确定答案」
```

### 2.6 强绑定 Rule
```
强绑定（必须遵守）：
- R2 NO_FABRICATION：追溯路径必须基于文件中实际存在的L1/L1.5内容
- R3 READ_FIRST：Step 1 必须完整读完三个文件，才能开始 Step 3 判断
- R6 ARTIFACT_FIRST：报告必须写文件，不能只在对话中输出

推荐注入：
- R1 EVIDENCE_FIRST：每个✅/❌ 都要有引用（「基于 P1『验证优先于感受』的约束」）
```

## 三、人层 vs Agent 层张力分析
```
人层「不言而喻」的假设：
1. 「能否追溯到L1.5/L1」——人知道「追溯」意味着什么，AI 需要知道
   「命题级别的语义对应」而非「词语重叠」
   修订：2.5 失败模式1 明确「追溯必须找到命题级别对应，不是词语重合」

2. 「漂浮任务」——人理解为「和认知根脱钩」，AI 可能理解为「不在清单里」
   修订：2.3 情况A 处理「弱追溯」，区分❌（无根）和⚠️（弱根）

AI 可能产生歧义的点：
1. L1.5 候选原则是否算「有效认知根」
   → 修订：Step 3.2 明确「候选原则也算，标注为候选级追溯」
```
