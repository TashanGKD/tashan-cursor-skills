## 四、接口协议

### 人层 API（REST）

| 方法 | 路径 | 功能 | 认证 |
|---|---|---|---|
| POST | /api/auth/register | 注册 | 无 |
| POST | /api/auth/login | 登录 | 无 |
| GET | /api/[资源] | 获取[资源]列表 | JWT/API Key |
| POST | /api/[资源] | 创建[资源] | JWT/API Key |
| GET | /api/[资源]/{id} | 获取单个[资源] | JWT/API Key |
| PUT | /api/[资源]/{id} | 更新[资源] | JWT/API Key |
| DELETE | /api/[资源]/{id} | 删除[资源] | JWT/API Key |

### 智能体层 API（扩展接口）

| 方法 | 路径 | 功能 | 说明 |
|---|---|---|---|
| GET | /api/[资源]/state | 获取状态（智能体可感知） | 结构化状态返回 |
| POST | /api/[资源]/actions | 触发操作（智能体可执行） | 批量操作接口 |

### SSE 流式接口（如有 AI 输出）

```
POST /api/chat/stream
Content-Type: application/json
Authorization: Bearer {token}

响应格式：
data: {"type": "thinking", "content": "..."}
data: {"type": "output", "content": "..."}
data: {"type": "done"}
data: [DONE]
```

### API Key 管理接口（双轨认证）

```
POST   /api/me/api-key   生成新 Key（返回明文，仅此一次）
GET    /api/me/api-key   查看 Key 信息（只返前缀，不返明文）
DELETE /api/me/api-key   吊销 Key
```

## 七、安全与权限

| 安全要求 | 实现方式 |
|---|---|
| 用户数据隔离 | 所有查询带 user_id 过滤 |
| API Key 安全 | 只存 SHA-256 哈希，不存明文 |
| JSONB 注入防护 | 使用参数化查询，CAST 语法 |
| 权限检查 | 每个路由 Depends(get_current_user) |

## 九、产品范式变更：废弃代码清单（**范式变更时必填**，功能叠加可跳过）

> **触发条件**（满足任一）：
> - 整体 UI 主题/风格发生根本性变化
> - 核心布局结构重写（如单栏→三栏；双模式→统一登录）
> - 某个旧功能模式在新产品里「概念上不存在」
> - 用户类型/权限模型发生变化

> **目的**：前端/后端开发在拿到技术架构时，能直接知道「哪些旧代码必须删除」，
> 不会因为不知道要清理而在旧代码上打补丁。

| 废弃的代码/模式 | 所在文件 | 操作 | 废弃原因 |
|---|---|---|---|
| 示例：IS_PROXY 访客判断逻辑 | App.tsx, LoginPage.tsx | **删除** | 新产品无访客模式，所有人统一登录 |
| 示例：旧深色背景 `#0d0d0f` | index.css | **替换** 为 TopicLab 浅色主题 | 品牌规范变更 |
| 示例：`?mode=proxy` URL 参数处理 | router | **删除** | 无访客链接概念 |
| [本项目具体废弃项] | [文件路径] | 删除/替换/保留 | [原因] |

**前端/后端开发激活前必读此清单。** 废弃操作不可省略，遗漏 = 旧代码残留 = 范式污染。
